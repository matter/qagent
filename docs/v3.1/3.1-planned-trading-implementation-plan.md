# QAgent V3.1 Planned Trading Model Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add an opt-in planned-price execution model across legacy backtest, legacy paper trading, and V3.0 StrategyGraph without changing existing `next_open` behavior.

**Architecture:** Implement a small shared execution helper first, then wire it into each runtime. REST, MCP, and UI expose configuration only after service-level behavior is covered by tests. Historical results and DB schema must remain compatible; new execution details are stored in existing JSON/config/metadata fields unless a later task proves a schema change is unavoidable.

**Tech Stack:** Python 3, FastAPI, DuckDB, pandas, unittest/pytest-compatible tests, React + TypeScript + Ant Design.

---

## 1. Implementation Principles

- Keep `next_open` as the default until a separate default-switch decision is approved.
- Add `planned_price` as an explicit execution mode.
- Use one shared fill-decision function everywhere. Do not duplicate the high/low buffer rule in separate services.
- Do not change strategy authoring requirements. `planned_price` is optional.
- Do not require DB rebuild. Existing backtest, signal, paper trading, and StrategyGraph rows must remain readable.
- Keep REST, MCP, and UI behavior consistent by routing through the same backend services.
- Commit after each milestone with a focused message.

## 2. Target Semantics

Execution mode:

```text
execution_model = "next_open" | "planned_price"
```

Planned-price fill condition:

```text
low * (1 + planned_price_buffer_bps / 10000)
  <= planned_price
  <= high * (1 - planned_price_buffer_bps / 10000)
```

Default:

```text
planned_price_buffer_bps = 50
```

Fallback:

```text
strategy planned_price missing/invalid -> decision_date close
```

Failed order:

```text
same-day cancel, no position/cash mutation
```

## 6. Data Migration and Compatibility

### 6.1 DB Schema

Initial implementation should avoid schema migrations.

Use existing fields:

- backtest config JSON
- backtest metrics/diagnostics JSON
- trade metadata JSON
- paper session config/metadata JSON
- StrategyGraph execution policy params JSON

Only add columns if a UI or API query needs indexed filtering by `execution_model`. If a column is required later, migration must:

- add nullable/defaulted columns
- backfill existing rows as `next_open`
- run dry-run row count before apply
- keep old JSON fields readable

### 6.2 Backward Compatibility

Compatibility requirements:

- Existing API requests without `execution_model` behave as `next_open`.
- Existing saved strategies with no `planned_price` column still run.
- Existing paper sessions still advance with open-price execution unless configured otherwise.
- Existing StrategyGraph execution policies remain valid.
- Existing backtest result readers tolerate missing planned diagnostics.

### 6.3 Result Comparability

Every new result must record:

```json
{
  "execution_model": "planned_price",
  "planned_price_buffer_bps": 50,
  "planned_fill_count": 82,
  "fallback_close_count": 12,
  "planned_fill_rate": 0.82,
  "fallback_close_rate": 0.12
}
```

This prevents accidental comparison between next-open and planned-price results without context.

## 7. Risk Register

| Risk | Impact | Mitigation |
| --- | --- | --- |
| high/low daily bar overstates fill probability | Backtest optimistic | buffer bps, explicit diagnostics, compare with next_open |
| missing strategy planned price leads to silent behavior | Results hard to interpret | fallback count and source required |
| paper trading and backtest diverge | Reproducibility loss | shared helper and shared tests |
| old result readers break on new fields | UI/API regression | optional response fields and default values |
| default switch changes historical behavior | User confusion | keep opt-in until separate approval |
| CN_A rules add complexity | False fills on suspended/limit stocks | run trade-status checks before planned-price fill |

## 8. Acceptance Criteria

- `next_open` remains default and existing tests for old execution pass.
- `planned_price` fills only inside buffered high/low range.
- successful planned trades use `planned_price` as accounting price.
- failed planned orders do not mutate cash or holdings.
- missing planned price uses decision close fallback and is counted.
- legacy backtest, paper trading, and StrategyGraph share the same helper.
- API, MCP, frontend types, and UI controls expose the same config names.
- Agent guide documents full workflow.
- No generated data, logs, DB files, or frontend build artifacts are committed.

## 9. Default-Switch Gate

Do not make `planned_price` the system default in this implementation.

A later change may switch the default only after collecting a comparison report with:

- at least one representative US universe
- at least one representative CN_A universe if CN planned trading is enabled
- same strategy under `next_open` and `planned_price`
- fill rate distribution
- fallback rate
- turnover impact
- final NAV and drawdown deltas
- top failure reasons
- human approval recorded in docs/backlog or a V3.1 report
