# QAgent V3.1 计划交易模型需求方案

> 文档状态：方案版
> 版本定位：V3.1 迭代
> 目标读者：开发 agent、研究 agent、系统维护者
> 相关上下文：QAgent 是本地优先、单用户、低频量化研究系统。REST、MCP、React UI 必须复用同一 backend service layer。

## 1. 背景

当前 legacy 回测、legacy paper trading 和 V3.0 StrategyGraph 的主要执行假设是：

- T 日收盘后基于可获得数据产生信号。
- T+1 日按开盘价成交。
- 成交价、目标股数和收益复现都锚定 T+1 open。

这个假设实现简单，但在真实研究和复盘中有两个问题：

1. 开盘价不一定可稳定锚定。真实执行可能无法在开盘点成交，导致策略表现难以复现。
2. 策略实际更常表达为“计划在某个可接受价格成交”，而不是“无条件开盘成交”。

V3.1 引入“计划交易模型”，把成交价从固定 open 切换为策略或系统生成的计划价格，并用 T+1 日线高低价区间判断是否可成交。

## 2. 目标

- 新增执行模式 `planned_price`，与现有 `next_open` 并存。
- 保持 T 日决策、T+1 日执行的时间语义，不引入未来函数。
- 允许策略输出计划成交价；策略没有输出时，系统可使用 T 日收盘价作为计划价 fallback。
- 回测、信号诊断、paper trading、StrategyGraph 使用同一套成交判定语义。
- 默认行为暂不改变。未显式选择 `planned_price` 的旧策略、旧回测、旧 paper session 仍按 `next_open` 运行。
- 所有计划价、fallback、成交失败原因和成交率必须可诊断、可审计。

## 3. 非目标

- 不接入真实券商下单。
- 不模拟分钟级盘口、排队、撮合顺序或 intraday path。
- 不立即把系统默认执行模式切到 `planned_price`。
- 不要求策略直接输出目标股数。目标股数仍由组合目标权重、组合净值、计划价和市场交易规则推导。
- 不在 V3.1 引入复杂订单生命周期。成交失败的计划单当日取消，不跨日挂单。

## 4. 核心语义

### 4.1 时间线

```text
T 日收盘后
  -> 使用截至 T 日收盘后可获得的数据
  -> 生成 T+1 日目标组合和调仓计划
  -> 每个订单包含 planned_price

T+1 日完成后
  -> 使用 T+1 日 daily bar 的 high/low 判断计划价是否可成交
  -> 成功则按 planned_price 记账
  -> 失败则订单取消，现金和持仓不变
```

### 4.2 订单字段

计划交易订单至少包含：

| 字段 | 说明 |
| --- | --- |
| `decision_date` | 产生信号和调仓计划的日期，通常为 T |
| `execution_date` | 计划执行日期，通常为 T+1 交易日 |
| `asset_id` 或 `ticker` | 交易标的，V3.0 使用 `asset_id`，legacy 使用 `ticker + market` |
| `side` | `buy` 或 `sell` |
| `planned_price` | 计划成交价 |
| `target_weight` | 目标权重，legacy 策略仍以权重为主 |
| `target_quantity` | 系统根据组合净值和计划价推导，策略不强制输出 |
| `execution_model` | `planned_price` |
| `planned_price_source` | `strategy_output`、`decision_close` 或后续扩展值 |

### 4.3 计划价来源

优先级：

1. 策略显式输出 `planned_price`。
2. 策略未输出、输出缺失或输出非正数时，系统使用 T 日收盘价作为 fallback。

fallback 必须写入诊断：

```json
{
  "planned_price_source": "decision_close",
  "planned_price_fallback": "decision_close",
  "fallback_reason": "missing_or_invalid_strategy_planned_price"
}
```

选择 T 日 close 作为 fallback 的原因是它符合“收盘后制定下一日计划”的低频研究语义，也不会读取 T+1 的数据。

### 4.4 成交判定

给定 T+1 日线：

- `low`: T+1 最低价
- `high`: T+1 最高价
- `planned_price`: 计划成交价
- `planned_price_buffer_bps`: 价格缓冲，默认 `50` bps，即 `0.5%`

成交成功条件：

```text
low * (1 + buffer) <= planned_price <= high * (1 - buffer)
```

默认值展开为：

```text
low * 1.005 <= planned_price <= high * 0.995
```

这意味着计划价需要落在日内区间中更保守的一段，而不是只要位于 `[low, high]` 就认为成交。

### 4.5 成交价格和成本

- 成交成功时，记账价格为 `planned_price`，不是 open、close 或区间中点。
- 交易成本沿用当前市场 profile 或 legacy 配置中的 commission、slippage、stamp tax、min commission 等模型。
- `slippage_rate` 是否继续叠加需要在实现中保持现有口径。V3.1 默认保留现有成本模型，不把 `planned_price_buffer_bps` 解释为成本。

### 4.6 成交失败

成交失败时：

- 不产生 trade row，或产生 `unfilled` 诊断记录但不改变持仓和现金。
- 订单当日取消，不进入下一交易日。
- 回测和 paper trading 的组合会因为未成交而偏离目标权重。
- 诊断需要记录失败原因：
  - `missing_bar`
  - `missing_high_low`
  - `planned_price_outside_buffered_range`
  - `not_trading`
  - `suspended`
  - `invalid_planned_price`
  - `insufficient_cash` 或已有交易规则阻断原因

## 5. 与现有系统的关系

### 5.1 Legacy Backtest

当前 `backend/services/backtest_engine.py` 的 `BacktestEngine.run()` 输入是目标权重矩阵，执行逻辑固定使用 T+1 open。

V3.1 应新增可选输入：

- `execution_model`: `next_open` 或 `planned_price`
- `planned_price_buffer_bps`: 默认 `50`
- `planned_prices`: 与 signals 同 index/columns 的计划价矩阵，可为空

`backend/services/backtest_service.py` 负责从策略原始信号中提取 `planned_price`，并在缺失时使用 T 日 close fallback。这样 `BacktestEngine` 仍只负责执行模拟，不负责理解策略源码。

### 5.2 Legacy Paper Trading

当前 `backend/services/paper_trading_service.py` 的 `_execute_trades_cached()` 使用 price cache 中的 open/close。

V3.1 需要让 paper trading：

- price cache 增加 high/low。
- session 或 advance config 支持 `execution_model`。
- `planned_price` 缺失时使用上一个决策日 close fallback。
- 记录计划单、成交成功、成交失败和 fallback 诊断。

Paper trading 是模拟交易，不是真实券商挂单。计划交易失败后不应自动重试，除非未来新增订单生命周期策略。

### 5.3 V3.0 StrategyGraph

当前 `backend/services/portfolio_assets_3_service.py` 只支持 `_SUPPORTED_EXECUTION = {"next_open"}`，`backend/services/strategy_graph_3_service.py` 的 `_execute_backtest_orders()` 按 `price_field` 加载价格。

V3.1 应新增 execution policy：

```json
{
  "policy_type": "planned_price",
  "params": {
    "planned_price_source": "strategy_or_decision_close",
    "planned_price_buffer_bps": 50,
    "fallback": "decision_close",
    "order_ttl": "same_day"
  }
}
```

StrategyGraph 的 order intent 应携带 `planned_price`、`planned_price_source` 和 `execution_model`。回测执行时加载 execution date 的 high/low，并复用共享成交判定函数。

### 5.4 REST / MCP / UI

新增能力必须先落在 service layer，再暴露到 REST、MCP 和 UI。

REST 和 MCP：

- 回测请求配置增加 `execution_model` 和 `planned_price_buffer_bps`。
- 旧请求缺省时仍视为 `next_open`。
- 任务结果和 backtest detail 增加执行诊断摘要。

UI：

- 回测面板增加执行模式选择：`Next Open`、`Planned Price`。
- Planned Price 模式显示 buffer bps 输入。
- 回测结果展示 fill rate、unfilled count、fallback count、主要失败原因。

Agent 文档：

- 更新 `docs/AGENT_GUIDE.md`，说明策略如何输出 `planned_price`、如何选择执行模式、如何检查诊断。

## 6. 策略接口建议

Legacy `StrategyBase.generate_signals(context)` 当前要求返回 indexed by ticker 的 DataFrame，至少包含：

- `signal`
- `weight`
- `strength`

V3.1 建议允许额外列：

- `planned_price`
- `planned_price_reason`

示例：

```python
signals.loc[ticker, "signal"] = 1
signals.loc[ticker, "weight"] = 0.05
signals.loc[ticker, "strength"] = score
signals.loc[ticker, "planned_price"] = float(context["close"].loc[ticker])
signals.loc[ticker, "planned_price_reason"] = "decision_close_limit"
```

策略不需要输出 `target_quantity`。系统会在执行阶段根据组合净值、目标权重、计划价和市场 lot size 推导数量。

## 7. 客观评判

### 7.1 价值

计划交易模型比固定开盘价更接近低频研究的实际执行过程：

- 策略可以明确表达“我愿意在什么价格交易”。
- 回测可以体现交易失败对组合漂移和收益的影响。
- 结果更容易解释给 human reviewer，因为成交失败有清晰原因。
- 后续可自然扩展到 limit order、volume cap、limit up/down、partial fill 等模型。

### 7.2 局限

该模型仍然是 daily bar 级别的近似：

- 日线 high/low 不能证明计划价在买卖方向上真实可成交。
- high/low 不包含成交量分布和排队信息。
- 对涨跌停、停牌、ST、最小交易单位、流动性不足等市场规则仍依赖现有 trade status 和 trading rules。
- 价格 buffer 是保守近似，不等价于真实滑点。

因此 V3.1 不应把 planned price 回测结果直接视为实盘可复制结果，而应作为比 next open 更可解释的研究执行模型。

### 7.3 推荐取舍

- 第一阶段保持 `next_open` 默认，`planned_price` 显式 opt-in。
- 所有新结果必须记录 `execution_model`，避免后续比较时混淆。
- 同一策略应支持 next_open 与 planned_price A/B 对照。
- 在 planned_price 成为默认前，需要至少检查：
  - fill rate
  - fallback rate
  - planned vs next_open 收益差异
  - 失败订单集中在哪些股票和日期
  - CN_A 的涨跌停、停牌和 lot size 影响

## 8. 验收标准

V3.1 需求完成后，应满足：

- 未配置 `execution_model` 的历史回测行为和结果不变。
- 选择 `planned_price` 后，成交价为计划价，成交成功由 high/low buffer 判定。
- 缺失计划价时使用 T 日 close fallback，并在诊断中计数。
- 成交失败不会改变现金和持仓。
- legacy backtest、legacy paper trading、V3.0 StrategyGraph 使用同一套成交判定函数。
- UI 能选择执行模式并展示计划交易诊断。
- REST/MCP/API 类型和 `docs/AGENT_GUIDE.md` 均包含 planned price 用法。
- 新增测试覆盖成功成交、失败成交、fallback、缺失行情、旧模式兼容和 StrategyGraph policy。
