# QAgent V3.1 Strategy Parameters and Execution Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move strategy-owned defaults and order execution intent into strategy metadata/output while keeping execution, accounting, diagnostics, and reproducibility owned by shared backtest/paper engines.

**Architecture:** Strategy code may declare default research parameters and per-order execution intent. Backtest and paper sessions merge those defaults with run/session overrides, then execute through shared service-layer helpers. The first shipped increment is `planned_price_fallback="next_close"` for legacy backtest and legacy paper trading; wider dynamic per-order execution is staged behind explicit schema and tests.

**Tech Stack:** Python 3, FastAPI, DuckDB, pandas, unittest, React + TypeScript + Ant Design.

---

## 1. Scope

This plan covers three related but separable upgrades:

1. Strategy default parameters: let strategies declare preferred sizing, rebalance, holding, and execution defaults.
2. Dynamic execution intent: let strategy output specify order-level execution model and planned price metadata.
3. Planned-price close fallback: when a planned-price order does not satisfy the buffered high/low range, optionally fill at the execution day's close.

The current implementation task is item 3. Items 1 and 2 are planned milestones because they touch API shape, UI workflow, docs, and reproducibility semantics.

## 2. Design Rules

- Preserve legacy compatibility. Missing new fields must behave exactly like the current system.
- Keep `next_open` as the default execution model until a separate default-switch decision is approved.
- Strategy code expresses intent and defaults; engines decide fills and mutate cash/positions.
- No same-day close execution for signals generated after the same close. If close execution is used, it means T decision, T+1 close execution.
- Persist effective configuration and diagnostics so agent and human reviewers can reproduce why an order filled, fell back, or failed.
- REST, MCP, UI, backtest, and paper must route through the same service behavior.

## 3. Configuration Model

Merge order:

```text
system defaults < strategy defaults < run/session override < per-order intent
```

Strategy default example:

```python
class MyStrategy(StrategyBase):
    name = "My Strategy"
    default_backtest_config = {
        "position_sizing": "raw_weight",
        "max_positions": 20,
        "rebalance_freq": "daily",
        "rebalance_buffer": 0.02,
        "min_holding_days": 3,
        "execution_model": "planned_price",
        "planned_price_buffer_bps": 50,
        "planned_price_fallback": "next_close",
        "constraint_config": {
            "max_single_name_weight": 0.08,
            "holding_period": {"min_days": 3, "max_days": 20},
        },
    }
```

Run/session config still owns experiment environment fields:

- `market`
- `universe_group_id`
- `start_date` / `end_date`
- `benchmark`
- `initial_capital`
- `commission_rate` / `slippage_rate`
- debug/replay flags

These can read strategy defaults as suggestions in the UI, but final values must be saved as `effective_config`.

## 4. Execution Model Semantics

Supported global modes:

- `next_open`: T decision, T+1 open fill.
- `planned_price`: T decision, T+1 planned-price fill test.
- `next_close`: planned future extension for T decision, T+1 close fill.

For this increment, `next_close` is only allowed as a planned-price fallback:

```json
{
  "execution_model": "planned_price",
  "planned_price_buffer_bps": 50,
  "planned_price_fallback": "next_close"
}
```

In 3.0 execution policy specs, the same execution-fill fallback is named `fill_fallback` because `fallback` is already used for the planned-price source fallback (`decision_close`):

```json
{
  "policy_type": "planned_price",
  "params": {
    "planned_price_buffer_bps": 50,
    "fallback": "decision_close",
    "fill_fallback": "next_close"
  }
}
```

Execution steps:

1. If planned price is valid and satisfies the buffered range, fill at `planned_price`.
2. If planned price is outside the buffered range and fallback is `next_close`, fill at T+1 close.
3. If close is missing or invalid, block the order.
4. If fallback is missing or `cancel`, keep the current cancel/no-fill behavior.

Diagnostics must distinguish:

- `fill_type="planned_price"`
- `fill_type="fallback_close"`
- `fill_type="blocked"`

Summary diagnostics should include counts and rates:

- `filled_order_count`
- `planned_fill_count`
- `fallback_close_count`
- `blocked_order_count`
- `planned_fill_rate`
- `fallback_close_rate`
- `blocked_rate`

If fallback close usage is high, reviewers should treat the strategy's planned-price quality as weak rather than as clean planned-order execution.

## 5. Dynamic Order Intent Future Schema

Strategy output may later include these optional columns:

```python
return pd.DataFrame(
    {
        "signal": [1],
        "weight": [0.08],
        "strength": [0.9],
        "execution_model": ["planned_price"],
        "planned_price": [101.5],
        "planned_price_buffer_bps": [50],
        "planned_price_fallback": ["next_close"],
        "time_in_force": ["day"],
        "reason": ["breakout limit entry"],
    },
    index=["AAPL"],
)
```

The engine should persist an order intent record in diagnostics with:

- `decision_date`
- `execution_date`
- `ticker` or `asset_id`
- `side`
- `target_weight`
- `execution_model`
- `planned_price`
- `planned_price_source`
- `fallback`
- `fill_type`
- `fill_price`
- `blocked_reason`
