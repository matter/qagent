# QAgent 3.0 生产可实施重构设计文档

> 分支：`3.0`  
> 基线：`origin/V2.0`  
> 文档状态：最终实施版  
> 实施原则：允许重建数据库和重写核心链路；保持本地优先、单用户、低频量化研究、REST/MCP/UI 共用服务层  
> 市场策略：US equities 先达到生产可研究深度；US/CN 双市场内核、表结构、时间模型、数据语义和交易规则从第一天内建

## 1. 总结

QAgent 3.0 的目标不是继续给 2.0 增加功能，而是重建一个可生产实施、可复现、可清理、可由 agent 自治研究的本地量化研究内核。

2.0 的主要问题不是单点 bug，而是抽象已经承载不了现有功能：

- `market` 横穿 DB、API、MCP 和 UI，US/CN 是横向硬接入，不是真正的市场模型。
- 策略接口把 alpha、选股、仓位、约束、执行、状态混在 `signal/weight/strength` 里。
- 模型训练内部临时拼接 feature/label/dataset，训练前数据不可审计，不利于 agent 自治研究。
- 信号生成被做成孤立模块，和回测、paper trading、模型预测存在口径漂移风险。
- 回测结果、诊断、临时实验、正式模型和策略混合保存，长期存储膨胀且难以清理。
- 缺少统一 `ResearchRun`、`Artifact`、`Lineage`、`PromotionPolicy` 和复现包，无法形成生产级研究闭环。

3.0 的核心方案：

- 建立 `ResearchProject + ResearchRun + Artifact + Lineage` 作为研究系统唯一事实来源。
- 建立 `MarketProfile + DataPolicy + TradingRuleSet + CostModel + BenchmarkPolicy`，避免 MarketProfile 变成新垃圾桶。
- 建立显式时间可用性模型：所有数据、特征、标签、预测、信号、成交都记录 `observation_date`、`available_at`、`decision_date`、`execution_date`。
- 把 `FeatureSet` 升级为 `FeaturePipeline`，把模型训练前的样本面板升级为一等 `Dataset`。
- 把模型记录拆成 `ModelExperiment` 和 `ModelPackage`。
- 把策略脚本拆成 `StrategyGraph`：`AlphaModel -> SelectionPolicy -> PortfolioConstruction -> RiskControl -> ExecutionPolicy -> StatePolicy`。
- 把组合构建、仓位、风险、成本、benchmark、执行、再平衡都做成独立可复用资产。
- 把 signal 改为各阶段输出能力，正式信号只来自 validated/published 策略图。
- 建立 `QuantQAGate`、`PromotionPolicy`、`AgentResearchPlan` 和 `ReproducibilityBundle`，防止 agent 自动过拟合和不可复现发布。
- 引入 `scratch / experiment / candidate / validated / published / archived` 生命周期，过程数据和正式资产分离保存。

3.0 第一阶段验收不是“页面能点通”，而是：

- US 日频研究链路从数据检查、因子、dataset、模型、策略图、回测、诊断、promote、正式信号、paper trading 全链路跑通。
- CN_A 的 market profile、数据语义、交易规则、可交易性字段、时间模型已经内建，不需要未来到处加 `market`。
- agent 能在系统内完成至少 5 条可执行研究流程，且每条流程最终优化到回测表现，并受 search budget、QA gate 和 promote policy 约束。

## 2. 目标、非目标和不变量

### 2.1 目标

1. 建立研究事实来源  
   所有研究活动都以 `ResearchRun` 为运行事实，以 `Artifact` 保存产物，以 `Lineage` 保存依赖。

2. 建立生产级时间正确性  
   所有计算必须知道数据何时观测、何时可用、何时决策、何时执行。

3. 建立双市场内核  
   US_EQ 和 CN_A 从 schema、profile、规则、provider、calendar、tradability 共同建模。US 先做深，CN 分阶段做完整。

4. 建立 agent-native 研究接口  
   agent 不只调用端到端任务，还能查询数据切片、profile dataset、解释回测日、比较 run、读取 artifact、执行 playbook。

5. 建立策略自由度  
   量化策略不被仓位、约束和 UI 控件过度限制。复杂策略可以自定义节点，但每层输出必须可解释、可复现。

6. 建立存储生命周期  
   临时研究过程可清理，正式发布资产可复现，系统长期运行不会被实验垃圾淹没。

### 2.2 非目标

- 不做实盘券商下单。
- 不做分钟、Tick、高频交易。
- 不做多用户权限体系。
- 不承诺免费 provider 数据达到商业生产质量。
- 不把 CN_A 在第一阶段做成完整可交易产品，但必须从第一天在架构和数据模型中真实存在。

### 2.3 系统不变量

这些规则不能被后续实现破坏：

- REST、MCP、UI 必须复用同一 service layer。
- 所有长任务必须走 `TaskExecutor`，并绑定 `ResearchRun`。
- 所有模型训练、策略回测、正式信号必须可追溯到 project、market profile、universe、data snapshot、代码版本和配置。
- 所有模型验证必须使用时间序列 split、rolling、expanding 或 purge-gap，不允许随机 K-Fold 用于市场时序。
- 所有策略和信号必须遵守时间可用性模型，不能使用决策日之后才可获得的数据。
- 正式信号只能由 `validated` 或 `published` 的策略图生成。
- 默认实验不能进入正式资产列表；必须经过 promote。
- cleanup 只能删除可重建或已降级的 artifact，不能破坏 summary、metrics、lineage 和复现索引。

## 3. 当前 2.0 诊断

### 3.1 可保留资产

2.0 有值得保留的工程基础：

- FastAPI、DuckDB、React、TaskExecutor、MCP 架构适合本地单用户研究。
- 后端 API 大体保持薄层，核心逻辑集中在 `backend/services/`。
- 因子和策略支持 Python 插件，符合量化研究习惯。
- 回测已坚持 T 日决策、T+1 open 执行。
- 模型训练已避免随机 K-Fold。
- 部分诊断能力已经出现，例如 signal diagnose、rebalance diagnostics。

### 3.2 必须修正的问题

1. 研究对象缺失  
   当前系统有服务，没有研究内核。因子评估、模型训练、回测、信号运行彼此独立保存，缺少统一 run/artifact/lineage。

2. 多市场是横向补丁  
   `market` 字段在表、API、MCP、UI 中横向传播。CN 资产、内置因子、默认股票池、benchmark 都像 US 流程翻译版。

3. 策略接口错位  
   当前 `StrategyBase.generate_signals(context)` 只能返回 `signal/weight/strength`，导致 alpha、selection、portfolio、risk、execution 混在策略源码和回测服务之间。

4. Dataset 不存在  
   模型训练内部临时构造 X/y，agent 无法在训练前审计样本覆盖、缺失、label 分布、特征漂移和泄漏风险。

5. FeaturePipeline 太弱  
   当前 preprocessing 接受 `neutralize` 概念但实际静默映射为 `None`，这会误导 human 和 agent。

6. 信号模块孤立  
   `SignalService` 复制回测链路一部分逻辑，长期会造成 signal、backtest、paper trading 口径差异。

7. 结果存储不可持续  
   大量 JSON 存 NAV、trades、diagnostics；临时实验和正式结果混在一起，存储会持续膨胀。

8. agent 会绕开系统  
   当前 MCP 工具偏端到端，缺少 dataset profile、artifact query、run explain、strategy day inspect 等中间能力。

9. 缺少 agent 优化边界  
   如果 agent 可以无限调参优化回测，系统会变成自动过拟合机器。

10. 缺少发布标准  
   当前 status/result_level 不足以表达“是否可以用于正式信号”。需要可执行的 QA gate 和 promotion policy。

## 4. 目标架构

### 4.1 分层

```text
REST / MCP / React UI
  -> Application Services
  -> Research Kernel
  -> Domain Engines
  -> Storage / Artifact Store / Provider Adapters
```

职责：

- REST/MCP/UI：请求形状、任务提交、展示和人机交互。
- Application Services：编排业务用例，例如创建 dataset、训练模型、跑回测、promote。
- Research Kernel：管理 project、run、artifact、lineage、lifecycle、QA、promotion、retention、reproducibility。
- Domain Engines：market、data、universe、factor、dataset、model、strategy、portfolio、risk、execution、backtest、signal、paper。
- Storage：DuckDB 结构化表、Parquet 面板 artifact、model files、code snapshots、reports。

### 4.2 唯一事实来源

3.0 的事实来源分层如下：

- `ResearchRun` 是运行事实来源。任何长任务或重要同步研究操作都生成 run。
- `Artifact` 是产物事实来源。大型结果、面板数据、诊断 trace、模型文件、图表数据都作为 artifact 保存。
- `Lineage` 是依赖事实来源。run 和 artifact 的输入输出关系都记录为有向边。
- 业务资产表保存可发现、可复用的资产元数据，例如 `datasets`、`model_packages`、`strategy_graphs`。

禁止模式：

- 禁止某个服务只把结果写入自己的结果表而不创建 run。
- 禁止大型时序结果只塞进 JSON summary。
- 禁止正式资产缺少 lineage。
- 禁止 REST 和 MCP 绕过 service layer 单独拼业务逻辑。

### 4.3 运行时标准结构

所有 run 必须包含：

- `run_id`
- `project_id`
- `market_profile_id`
- `run_type`
- `status`
- `lifecycle_stage`
- `retention_class`
- `input_refs`
- `output_refs`
- `params_snapshot`
- `code_snapshot_refs`
- `data_snapshot_refs`
- `stage_timings`
- `metrics_summary`
- `qa_summary`
- `warnings`
- `error`

所有 artifact 必须包含：

- `artifact_id`
- `run_id`
- `artifact_type`
- `uri`
- `format`
- `schema_version`
- `byte_size`
- `content_hash`
- `lifecycle_stage`
- `retention_class`
- `cleanup_after`
- `rebuildable`
- `metadata`

## 5. 时间可用性模型

3.0 必须把时间正确性做成一等模型，而不是散落在注释和约定里。

### 5.1 时间字段

所有数据、特征、标签、预测、信号、订单、成交至少区分：

- `observation_date`: 市场或事件发生日期。
- `available_at`: 系统认为该数据可被使用的时间。
- `as_of_date`: 本次计算的数据快照截止日期。
- `decision_date`: 策略做决策的日期。
- `execution_date`: 订单或回测成交日期。
- `recorded_at`: 系统写入时间。

示例：

- 日线 close 的 `observation_date` 是 T，低频研究中 `available_at` 默认可设为 T 收盘后。
- T 日产生的信号，US 默认 T+1 open 执行。
- 财报、指数成分、停牌状态等未来数据必须有真实或保守的 `available_at`，不能只按 observation date 使用。

### 5.2 数据可用性规则

任何 run 在构建输入时必须满足：

```text
input.available_at <= decision_timestamp
```

标签是例外：label 可使用未来收益，但只能用于训练和评估，不能进入策略决策输入。Dataset 必须显式标注哪些列是 feature，哪些列是 label。

### 5.3 QA 检查

`QuantQAGate` 必须包含：

- feature available_at 检查。
- label horizon 与 split purge-gap 检查。
- 模型训练区间和回测区间重叠检查。
- universe membership point-in-time 检查。
- benchmark 和 market profile 一致性检查。
- execution_date 是否是有效交易日检查。
- CN_A 涨跌停、停牌、ST、上市天数等可交易性检查。

## 6. 市场和数据语义设计

### 6.1 拆分 MarketProfile

为了避免 `MarketProfile` 变成新垃圾桶，3.0 将市场能力拆为五类对象：

- `MarketProfile`: 市场身份、资产命名、默认规则引用。
- `DataPolicy`: provider、价格复权语义、字段解释、数据质量等级。
- `TradingRuleSet`: 日历、T+1/T+0、涨跌停、停牌、ST、手数、可交易性。
- `CostModel`: commission、slippage、tax、市场默认成本。
- `BenchmarkPolicy`: 默认 benchmark、可用 benchmark、指数数据语义。

`ResearchProject` 引用这些 policy，可以使用市场默认值，也可以在研究中显式覆盖。

### 6.2 US_EQ

US_EQ 第一阶段完整实现：

- symbol：标准 US ticker，例如 `AAPL`。
- provider：yfinance，数据等级默认 `exploratory`。
- calendar：NYSE/Nasdaq 日历。
- price semantics：明确 adjusted/unadjusted 字段。
- benchmark：SPY、QQQ。
- trading rules：T 日决策，T+1 open 执行。
- tradability：缺价、stale ticker、成交量为 0、退市风险标记。

### 6.3 CN_A

CN_A 从第一天进入 schema 和规则测试，分阶段实现数据接入：

- symbol normalization：`600000.SH`、`000001.SZ`、provider ticker 映射。
- calendar：SSE/SZSE 交易日历。
- provider：BaoStock 起步，保留 Tushare/本地数据扩展。
- price adjustment：前复权、后复权、不复权必须显式。
- tradability：停牌、ST、涨跌停、一字板、上市天数、退市。
- execution：股票 T+1 卖出限制、涨跌停不可成交、最小交易单位。
- universe：全 A、沪深 300、中证 500、中证 1000、板块/行业/市值过滤。

CN_A 不再通过 `cn_` 前缀复制 US 因子和标签。FactorSpec 可以声明适用市场，FactorRun 绑定 market profile。

## 7. 核心领域对象

### 7.1 ResearchProject

项目是人和 agent 的研究上下文，替代 UI 全局 market selector。

关键字段：

- `id`
- `name`
- `market_profile_id`
- `default_universe_id`
- `data_policy_id`
- `trading_rule_set_id`
- `cost_model_id`
- `benchmark_policy_id`
- `artifact_policy_id`
- `created_at`

### 7.2 Asset

资产使用稳定 `asset_id`，ticker 只是 identifier。

关键表：

- `assets(asset_id, market_profile_id, symbol, display_symbol, name, exchange, sector, industry, status, metadata)`
- `asset_identifiers(asset_id, identifier_type, identifier_value, valid_from, valid_to)`
- `asset_lifecycle(asset_id, listed_date, delisted_date, status_history)`

### 7.3 MarketDataSnapshot

数据更新不只写 daily bars，还要形成 snapshot。

关键字段：

- `snapshot_id`
- `market_profile_id`
- `provider`
- `data_policy_id`
- `as_of_date`
- `created_at`
- `coverage_summary`
- `quality_summary`

run 引用 snapshot，保证结果可解释。

### 7.4 Universe

Universe 是一等资产。

类型：

- static basket。
- filter expression。
- index constituents。
- materialized snapshot。

必须支持：

- `universe.materialize(start, end)` 返回 `date, asset_id, membership_state`。
- PIT 扩展位。
- CN_A 可交易性过滤。
- universe profile：日期覆盖、日均成分数、缺失、行业/市值分布。

### 7.5 FactorSpec 和 FactorRun

FactorSpec：

- code snapshot。
- params schema。
- required inputs。
- compute mode：`time_series`、`cross_sectional`、`panel`。
- expected warmup。
- applicable market profiles。
- semantic tags。

FactorRun：

- project。
- factor spec version。
- universe。
- date range。
- market data snapshot。
- data availability policy。
- output artifact。

### 7.6 FeaturePipeline 和 Dataset

FeaturePipeline 是有向 transform 图，不是简单 factor list。

节点类型：

- raw factor。
- lag。
- fill。
- winsorize。
- rank。
- zscore。
- neutralize。
- clip。
- interaction。
- market-aware transform。

Dataset 是模型训练和 agent 分析的一等资产。

Dataset 必须记录：

- project、market profile、universe。
- date range。
- feature pipeline version。
- label spec version。
- split policy。
- row index：`date, asset_id`。
- feature artifact。
- label artifact。
- dataset profile。
- leakage check result。

模型训练必须基于 dataset，不允许训练服务内部临时拼接不可审计的 X/y。

### 7.7 LabelSpec 和 LabelRun

LabelSpec 必须显式描述：

- target type。
- horizon。
- benchmark。
- future window semantics。
- tail NaN policy。
- path-aware 规则。
- tradability handling。

LabelRun 绑定 dataset/universe/date range，并保存 label artifact 和 profile。

### 7.8 ModelExperiment 和 ModelPackage

ModelExperiment 是研究运行，ModelPackage 是可引用资产。

ModelExperiment 保存：

- dataset id。
- split policy。
- objective。
- params。
- random seed。
- metrics。
- prediction artifact。
- feature importance。
- QA result。

ModelPackage 只能由 experiment promote 得到：

- frozen feature schema。
- frozen preprocessing state。
- model file artifact。
- prediction contract。
- source experiment id。
- promotion record。

### 7.9 StrategyGraph

StrategyGraph 是策略定义资产，不等于单个 Python 类。

标准节点：

```text
UniverseFilter
  -> AlphaModel
  -> SelectionPolicy
  -> PortfolioConstruction
  -> RiskControl
  -> ExecutionPolicy
  -> StatePolicy
```

每个节点必须声明：

- input schema。
- output schema。
- state usage。
- data requirements。
- params schema。
- code/config snapshot。
- explain payload schema。

允许节点类型：

- built-in config。
- Python plugin。
- factor reference。
- model package reference。
- composite graph。

legacy strategy adapter 保留，但只能作为迁移兼容层，不能作为 3.0 新策略主协议。

### 7.10 Portfolio、Risk、Cost、Benchmark 资产

组合构建和仓位必须拆出策略模块。

新增资产：

- `PortfolioConstructionSpec`
- `RiskControlSpec`
- `RebalancePolicySpec`
- `ExecutionPolicySpec`
- `StatePolicySpec`
- `CostModel`
- `BenchmarkPolicy`

这允许 agent 在同一 alpha 上独立优化组合方式，也允许同一组合规则复用于多个策略。

### 7.11 Signal

Signal 不再是孤立业务模块，而是 stage output。

类型：

- `FactorSignal`
- `ModelSignal`
- `StrategySignal`
- `ProductionSignalRun`

正式 signal 要求：

- 来源 StrategyGraph 必须是 `validated` 或 `published`。
- 模型、因子、dataset、portfolio/risk/execution policy 必须冻结。
- 必须输出 target portfolio、order intent、human summary 和 machine-readable details。

## 8. StrategyGraph Runtime

### 8.1 Runtime 流程

```text
build_context(decision_date)
  -> compute_alpha
  -> select_assets
  -> construct_portfolio
  -> apply_risk_control
  -> generate_order_intents
  -> execute_or_emit
  -> persist_stage_artifacts
```

### 8.2 输出对象

- `AlphaFrame(date, asset_id, score, rank, confidence, reason)`
- `SelectionFrame(date, asset_id, action, reason, score_ref)`
- `PortfolioTarget(date, asset_id, target_weight, cash_weight, construction_reason)`
- `ConstraintTrace(date, asset_id, before_weight, after_weight, rule_id, reason)`
- `OrderIntent(decision_date, execution_date, asset_id, side, target_weight, estimated_value, reason)`
- `ExecutionResult(execution_date, asset_id, filled, fill_price, cost, failure_reason)`

### 8.3 Backtest、Signal、Paper 统一

三者共享 runtime，区别只在最后：

- backtest：历史撮合，产出 fills、NAV、trades、diagnostics。
- production signal：不撮合，产出 target portfolio 和 order intent。
- paper：读取 session state 增量推进。

不允许 backtest、signal、paper 各自重写策略依赖、因子、模型、仓位、约束逻辑。

## 9. Agent 自治研究设计

### 9.1 AgentResearchPlan

agent 每次批量研究必须创建 `AgentResearchPlan`。

字段：

- `plan_id`
- `project_id`
- `hypothesis`
- `search_space`
- `budget`
- `stop_conditions`
- `validation_policy`
- `promotion_policy_id`
- `created_by`

### 9.2 自动优化边界

为了防止自动过拟合，agent 优化必须受约束：

- 参数空间必须显式声明。
- 最大 run 数、最大 wall time、最大数据范围必须限制。
- 至少一个 out-of-time 验证窗口。
- 参数 sweep 结果必须按 validation metric 排序，而不是只看训练/开发窗口。
- 禁止在发现 test 结果后继续用同一 test window 迭代。
- 每次修改必须标注修改层级：factor、feature、model、portfolio、risk、execution。

### 9.3 内置可执行研究流程

3.0 至少内置 6 条 playbook，每条都能最终优化到回测表现。

#### A. 单因子假设到因子策略回测

1. data/universe profile。
2. factor preview。
3. factor materialize。
4. factor evaluate。
5. factor signal。
6. top N strategy graph。
7. portfolio variant backtest。
8. QA gate。

目标：IC 稳定、分组收益、回测收益风险比、换手可控。

#### B. 多因子 Dataset 到学习排序模型回测

1. feature pipeline。
2. dataset materialize/profile。
3. model experiment。
4. prediction profile。
5. model package candidate。
6. model-driven strategy graph。
7. backtest compare。

目标：out-of-time RankIC、top/bottom spread、回测 Sharpe、最大回撤。

#### C. 组合构建和仓位规则优化

1. 固定 alpha。
2. 创建多种 portfolio construction。
3. 批量 construct target。
4. backtest compare。
5. attribution constraint drag。

目标：同等 alpha 下提升收益风险比、降低换手和集中度。

#### D. Walk-forward 鲁棒性研究

1. rolling/expanding split。
2. 多窗口训练。
3. 多窗口回测。
4. worst-window analysis。
5. stability report。

目标：降低单窗口过拟合，提升最差窗口表现。

#### E. Regime 和 Universe 条件研究

1. 构造 regime factor。
2. 构造多个 universe。
3. 分 regime factor evaluate。
4. 策略加 regime gate。
5. backtest attribution。

目标：识别策略适用市场状态，降低回撤。

#### F. 回测失败诊断和低自由度修复

1. inspect rebalance/trades。
2. 查询 stage artifacts。
3. 定位 alpha/selection/portfolio/risk/execution 层级。
4. 小范围修改对应节点。
5. compare before/after。

目标：减少异常换手、执行失败、约束拖累，不随意重写策略。

## 10. Quant QA Gate 和 PromotionPolicy

### 10.1 QA Gate

每个重要 run 自动生成 QA。

检查项：

- 数据覆盖率。
- stale/missing price。
- feature availability。
- label horizon 和 purge gap。
- split 合规。
- universe coverage。
- factor value NaN/inf。
- prediction distribution。
- turnover。
- cost sensitivity。
- max drawdown。
- benchmark consistency。
- extreme rebalance days。
- execution failure。
- CN_A tradability rules。

QA 输出：

- `pass`
- `warning`
- `fail`
- structured findings。
- blocking reason。

### 10.2 PromotionPolicy

promote 不能只看收益。默认 policy：

- 数据覆盖率达到阈值。
- 无 blocking leakage。
- out-of-time 指标过线。
- 回测最差年度或最差窗口不过度失效。
- 最大回撤低于阈值。
- 年化换手低于阈值或有解释。
- 成本敏感性可接受。
- 关键 artifact 和 lineage 完整。
- 策略图依赖的 model package、factor spec、portfolio/risk/execution policy 都已冻结。

Promote 输出 `PromotionRecord`：

- source run/artifact。
- policy id。
- passed checks。
- failed non-blocking warnings。
- approver：agent 或 human。
- promoted asset ids。

### 10.3 生命周期

状态：

- `scratch`: preview、小样本、短 TTL。
- `experiment`: 正常实验，大 artifact 可过期。
- `candidate`: 值得复查，不自动删。
- `validated`: 通过 QA，可用于 paper。
- `published`: 正式资产，可生成 production signal。
- `archived`: 不活跃，保留摘要和 lineage。

默认保留策略：

- scratch: 7-14 天。
- experiment large artifacts: 30-90 天。
- candidate: 不自动删。
- validated/published: 不自动删。
- archived: 保留 summary、metrics、lineage，可删除大 artifact。

cleanup 必须先执行 `cleanup_preview`。

## 15. 迁移方案

### 15.1 迁移策略

不强行兼容 2.0 schema。采用 side-by-side 迁移：

1. 备份 2.0 数据库和 artifacts。
2. 初始化 3.0 schema。
3. 迁移 US 核心数据。
4. 将可映射资产转换为 3.0 draft/candidate。
5. 将不可映射资产 archive。
6. 运行迁移对账。
7. 只在 3.0 链路上产生新正式资产。

### 15.2 迁移映射

- `stocks` -> `assets`、`asset_identifiers`。
- `daily_bars` -> `daily_bars` + `market_data_snapshots`。
- `stock_groups` -> `universes`。
- `factors` -> `factor_specs`。
- `factor_values_cache` -> `factor_runs` + artifact 或 `factor_values`。
- `feature_sets` -> `feature_pipelines`。
- `label_definitions` -> `label_specs`。
- `models` -> `model_experiments`，人工或 QA 后 promote 为 `model_packages`。
- `strategies` -> legacy adapter `strategy_graphs`。
- `backtest_results` -> archived `backtest_runs` summary。
- `signal_runs` -> archived `production_signal_runs` 或 experiment signal artifacts。
- `paper_trading_sessions` -> archive，必要时新建 3.0 paper session。

CN 2.0 资产只迁移到 CN_A archive/candidate，不作为 3.0 第一阶段可用性基线。

### 15.3 迁移验收

- 表级 row count 对账。

- universe 成员数。
- factors 数量和代码 hash。
- models 文件存在性。
- strategies 代码 hash。

## 19. 关键取舍

### 保留

- FastAPI、DuckDB、React、TaskExecutor、MCP。
- 本地优先、单用户。
- Python 插件能力。
- T+1 open 低频执行语义。
- 时间序列验证原则。

### 改造

- 从服务模块驱动改为研究内核驱动。
- 从 global market 改为 project + market policies。
- 从 feature set 改为 feature pipeline + dataset。
- 从 model record 改为 experiment/package。
- 从 strategy script 改为 strategy graph。
- 从 signal service 孤岛改为 stage output + production signal。
- 从永久堆结果改为 lifecycle + retention + promote。

### 明确不做

- 不继续修补 2.0 CN 横向接入体验。
- 不继续在旧策略接口叠仓位参数。
- 不允许 agent 无预算自动调参。
- 不把临时实验默认变成正式资产。

## 20. 结语

QAgent 3.0 的核心不是“更大的功能清单”，而是把研究系统的底层秩序建立起来。

真正要解决的是：

- 数据什么时候可用。
- 实验从哪里来。
- 结果如何复现。
- 策略每一层在做什么。
- agent 优化是否可信。
- 市场差异是否被正确建模。
- 临时过程和正式资产是否能长期共存。

只要这些问题在 3.0 内核中被压实，US equities 会先形成一条专业、可解释、可复现的低频量化研究闭环；CN_A 后续接入也会是市场 profile、数据语义和交易规则的自然扩展，而不是又一次横向硬补丁。
