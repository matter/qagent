# QAgent 3.0 分阶段实现路线图

> 对应设计文档：[3.0-refactor-design.md](./3.0-refactor-design.md)  
> 目标：把 3.0 重构拆成可稳定迭代、可验收、可回滚的工程里程碑  
> 原则：每个阶段都必须有清晰交付物、验收标准、兼容策略和风险边界

## 1. 实施总原则

3.0 是大规模重构，但不能用“大爆炸式改写”推进。推荐采用 side-by-side 架构：保留 2.0 可运行链路，同时逐步建设 3.0 内核；新能力优先走 3.0 service，旧能力通过 adapter 迁移。

核心原则：

- 每个里程碑都能独立合并和验收。
- 每个里程碑都不依赖完整 3.0 才能验证。
- 所有长任务新增能力必须绑定 `ResearchRun`。
- 迁移脚本先 dry-run，再真实写入。
- 先建不可变基础对象，再迁移业务链路。
- agent API 和 UI 不直接访问 DB，必须走 service layer。
- 每个阶段保留旧链路 fallback，直到对应 3.0 链路验收通过。

## 2. 里程碑总览

| 里程碑 | 名称 | 目标 | 可验收结果 |
| --- | --- | --- | --- |
| M0 | 冻结与基线保护 | 保护 2.0 数据和行为 | 备份、dry-run、基线 e2e 可运行 |
| M1 | Research Kernel | 建立 run/artifact/lineage/lifecycle | 新任务能产生 `task_id + run_id` |
| M2 | Market/Data Foundation | 建立 project、market policies、asset_id、snapshot | US 数据可查询，CN_A profile 可查询 |
| M3 | Migration Layer | 2.0 资产映射到 3.0 | 迁移报告可对账，legacy adapter 可跑 |
| M4 | Universe + Dataset | 建立训练前数据资产 | Dataset 可 materialize/profile/query |
| M5 | Factor Engine 3.0 | 因子 spec/run/artifact 化 | preview/materialize/evaluate 可运行 |
| M6 | Model Experiment | 模型训练基于 dataset | experiment/package/promote 初版可用 |
| M7 | Portfolio/Risk/Execution | 仓位、约束、执行资产化 | 同一 alpha 可比较组合构建 |
| M8 | StrategyGraph Runtime | 统一 backtest/signal/paper 运行时 | 单日 explain 和回测可用 |
| M9 | QA + Agent Playbooks | 防过拟合和自治研究流程 | 6 条 playbook 可执行，有 budget |
| M10 | Production Signal + Paper | 正式信号和 paper 迁入新 runtime | published graph 才能生成正式信号 |
| M11 | UI 工作台 | 从模块 UI 转项目/结果审阅 UI | human 可审阅、promote、cleanup |
| M12 | CN_A 分阶段接入 | 按 profile 接入 A 股能力 | CN data/universe/factor subset 可用 |

## 2.1 跨阶段实现契约

以下契约从 M1 开始生效，后续里程碑不能重新定义：

- 所有新长任务返回值必须同时包含 `task_id` 和 `run_id`。
- 所有新业务 API 优先接收 `project_id`；旧 `market` 参数只通过 adapter 转成默认 project。
- M1 必须创建 bootstrap project。即使 M2 才完整实现 market/data policies，M1 的 `ResearchRun.project_id` 也不能为空。
- 所有 artifact 必须有 `artifact_id`、`run_id`、`artifact_type`、`uri`、`content_hash`、`lifecycle_stage`、`retention_class`。
- 所有涉及数据时点的 artifact 或结构化表必须能表达 `observation_date`、`available_at`、`as_of_date`、`decision_date`、`execution_date` 中相关字段。
- 任何 promote 动作都必须产生 `PromotionRecord`；M6 可以先实现手动/轻量 promote，M9 再接入完整 `PromotionPolicy`。
- 任何 cleanup 动作必须先有 `cleanup_preview`，不能直接删除。
- REST、MCP、UI 不允许绕过 service layer 直接拼业务逻辑。

## 3. M0：冻结与基线保护

### 目标

在开始重构前，保护 2.0 数据、行为和开发节奏。避免实施过程中无法判断问题来自重构还是旧系统不稳定。

### 范围

- 锁定 3.0 分支。
- 备份 DuckDB、model files、custom factors、custom strategies、paper sessions。
- 固定当前 2.0 e2e 行为。
- 增加迁移 dry-run 框架，但不写入 3.0 schema。

### 交付物

- `scripts/backup_data.sh` 可执行备份。
- `scripts/migrate_3_0.py --dry-run` 初版。
- 2.0 baseline report。
- 当前 US 主链路 smoke test 记录。

### 验收

- 能从备份恢复 2.0 数据。
- `uv run python scripts/e2e_demo.py` 仍可运行，或明确记录现有失败。
- dry-run 能输出表级 row count，不改 DB。

### 风险和回滚

- 风险：重构中污染本地数据。
- 回滚：恢复备份，切回 2.0 schema，禁用 3.0 初始化。

## 4. M1：Research Kernel

### 目标

先建立 3.0 的事实来源：`ResearchRun`、`Artifact`、`Lineage`、生命周期和 artifact store。后续所有新功能都挂在这个内核上。

### 范围

- 新增 core schema。
- 创建 bootstrap project：`US Research`，绑定临时 `US_EQ` profile stub。
- 新增 `ResearchKernelService`。
- TaskExecutor 支持 `run_id`。
- artifact 写入本地目录和 DB metadata。
- lifecycle stage 和 retention policy 初版。

### 交付物

- 表：`research_projects`、`research_runs`、`artifacts`、`lineage_edges`、`artifact_retention_policies`。
- 服务：`backend/services/research_kernel_service.py`。
- API/MCP：`run.get`、`run.list`、`artifact.get`、`lineage.get`。
- artifact 路径规范：`data/artifacts/{scratch,experiments,candidates,published,archived}`。

### 验收

- 系统初始化后一定存在一个默认 project，可供 M1 run 绑定。
- 任意测试任务可创建 run。
- run 可记录 stage、status、params、warning、error。
- artifact 可写入、查询、计算 hash。
- lineage 可记录 run -> artifact。

### 兼容策略

2.0 任务表继续保留；新增任务先双写 `task_runs` 和 `research_runs`。旧 API 未传 `project_id` 时，统一落到 bootstrap project。

## 5. M2：Market/Data Foundation

### 目标

替换全局 `market` 思维，建立 project-scoped market policies 和稳定 `asset_id`。

### 范围

- `MarketProfile` 拆分为 `DataPolicy`、`TradingRuleSet`、`CostModel`、`BenchmarkPolicy`。
- 建立 `assets`、`asset_identifiers`、`daily_bars`、`trade_status`、`market_data_snapshots`。
- US_EQ 完成基础查询。
- CN_A 建立 profile fixture 和 tradability schema。

### 交付物

- 表：market/data 相关 schema。
- 服务：`MarketProfileService`、`AssetService`、`MarketDataService`。
- API/MCP：`project.create`、`market.list_profiles`、`market.get_profile`、`data.status`、`data.query_bars`。
- 默认 project：`US Research`。

### 验收

- 可通过 `project_id` 查询 US 数据状态。
- CN_A profile 可查询，包含 symbol、calendar、trading rules、tradability 字段。
- 新服务内部使用 `asset_id`，不再以 `market + ticker` 作为主要业务键。

### 兼容策略

旧 API 仍接受 `market`，通过 adapter 转成默认 project。

## 6. M3：Migration Layer

### 目标

把 2.0 核心资产迁移或归档到 3.0，同时给旧策略、旧模型、旧因子提供 adapter。

### 范围

- 迁移 US stocks/daily_bars/groups/factors/feature_sets/labels/models/strategies/backtests。
- CN 资产只迁到 archive/candidate，不作为第一阶段验收基线。
- legacy strategy adapter。
- legacy factor adapter。

### 交付物

- `scripts/migrate_3_0.py` 支持 dry-run 和 apply。
- `migration_reports/` 输出对账报告。
- legacy adapter 服务。

### 验收

- 迁移报告包含 row count、hash、失败列表。
- 一个 migrated factor 可以 preview。
- 一个 migrated universe 可以 materialize。
- 一个 legacy strategy 可以通过 adapter 跑一次 3.0 backtest smoke。

### 回滚

迁移采用新 schema 写入，不覆盖 2.0 表；失败时删除 3.0 schema 或重建 DB。

## 7. M4：Universe + Dataset Engine

### 目标

把训练前样本面板变成一等资产，先解决模型训练不可审计的问题。

### 范围

- Universe materialization。
- FeaturePipeline 基础节点。
- LabelSpec/LabelRun。
- Dataset materialize/profile/sample/query。
- leakage QA 初版。

### 交付物

- 表：`universes`、`universe_memberships`、`feature_pipelines`、`feature_pipeline_nodes`、`label_specs`、`label_runs`、`datasets`、`dataset_profiles`。
- 服务：`UniverseService`、`FeaturePipelineService`、`DatasetService`。
- API/MCP：`universe.materialize`、`dataset.create`、`dataset.materialize`、`dataset.profile`、`dataset.sample`。

### 验收

- agent 可在训练前查询 dataset coverage、missing、label distribution。
- Dataset artifact 可复用，不需要每次模型训练重新拼 X/y。
- QA 能阻断明显 feature/label 日期错位。

### 兼容策略

旧 `FeatureService` 暂保留；新模型训练优先使用 Dataset，旧模型训练入口标记 legacy。

## 8. M5：Factor Engine 3.0

### 目标

把因子从 CRUD + cache 升级为 spec/run/artifact/lineage 模型，支持 preview 和未来横截面因子。

### 范围

- FactorSpec/FactorRun 分离。
- preview/validate/materialize/evaluate。
- time_series legacy adapter。
- cross_sectional skeleton。
- factor artifact 写入。

### 交付物

- 表：`factor_specs`、`factor_runs`、`factor_values`、`factor_signals`。
- 服务：`FactorSpecService`、`FactorRunService`、`FactorEngine3`。
- API/MCP：`factor.preview`、`factor.materialize`、`factor.evaluate`、`factor.compare`。

### 验收

- 因子 preview 不创建正式资产，只产生 scratch run。
- materialize 产出 FactorRun 和 artifact。
- evaluate 产出 QA、metrics、lineage。
- migrated factor 可通过 legacy adapter 运行。

## 9. M6：Model Experiment + Package

### 目标

模型训练必须基于 Dataset，并区分实验和可发布模型包。该阶段只实现轻量 promote：人工确认或基础 QA 通过后生成 `ModelPackage` 和 `PromotionRecord`；完整策略化 `PromotionPolicy` 在 M9 接入。

### 范围

- ModelExperiment。
- PredictionRun。
- ModelPackage。
- ModelPackage promote 初版，必须产生 `PromotionRecord`。
- prediction profile。

### 交付物

- 表：`model_specs`、`model_experiments`、`prediction_runs`、`model_packages`、`model_signals`。
- 服务：`ModelExperimentService`、`ModelPackageService`。
- API/MCP：`model.train_experiment`、`model.compare_experiments`、`model.promote`、`model.predict_panel`。

### 验收

- 同一 Dataset 上可训练多个 experiment 并比较。
- experiment 记录 dataset id、split policy、params、metrics、prediction artifact。
- 只有 promoted package 可作为稳定策略依赖。
- M6 的 promote 可以是手动/基础 QA，但不能绕过 `PromotionRecord`。

### 兼容策略

旧 `models` 表迁移为 experiment；是否变成 package 由 QA 或 human 决定。

## 10. M7：Portfolio/Risk/Execution Assets

### 目标

把仓位、组合、风险、执行从策略模块拆出来，建立可复用、可比较、可解释的资产。

### 范围

- PortfolioConstructionSpec。
- RiskControlSpec。
- RebalancePolicySpec。
- ExecutionPolicySpec。
- StatePolicySpec。
- CostModel 引用。

### 交付物

- 表：`portfolio_construction_specs`、`risk_control_specs`、`rebalance_policy_specs`、`execution_policy_specs`、`state_policy_specs`、`portfolio_runs`。
- 服务：`PortfolioService`、`RiskService`、`ExecutionPolicyService`。
- 内置规则：equal weight、score proportional、inverse vol、max single weight、max positions、turnover budget、rebalance band。

### 验收

- 同一 alpha frame 可生成多种 portfolio target。
- constraint trace 能解释权重被修改的原因。
- agent 可单独比较组合构建器，不需要改策略源码。

## 11. M8：StrategyGraph Runtime

### 目标

实现 3.0 策略运行时，统一 backtest、production signal、paper trading 的策略口径。

### 范围

- StrategyGraph schema。
- node protocol。
- runtime context。
- stage artifacts。
- simulate_day/explain_day。
- legacy strategy adapter。
- backtest 初版迁入 runtime。

### 交付物

- 表：`strategy_graphs`、`strategy_nodes`、`strategy_signals`、`backtest_runs`、`backtest_daily`、`backtest_trades`。
- 服务：`StrategyGraphService`、`StrategyRuntimeService`、`BacktestService3`。
- API/MCP：`strategy.create_graph`、`strategy.simulate_day`、`strategy.explain_day`、`backtest.run`。

### 验收

- 单日 explain 返回 alpha、selection、portfolio target、constraint trace、order intent。
- backtest 产出 structured daily/trades 和 artifact。
- legacy strategy adapter 可跑。
- runtime 不允许 backtest/signal/paper 各自复制策略逻辑。

## 12. M9：QA Gate + Agent Playbooks

### 目标

让 agent 自治研究可控、可审计，避免自动过拟合。

### 范围

- QuantQAGate。
- 完整 PromotionPolicy，并接管 M6 的轻量 promote。
- AgentResearchPlan。
- search budget。
- stop conditions。
- 6 条 playbook。

### 交付物

- 表：`qa_reports`、`promotion_policies`、`promotion_records`、`agent_research_plans`。
- 服务：`QAGateService`、`PromotionService`、`AgentPlaybookService`。
- API/MCP：`qa.get_report`、`run.compare`、`artifact.promote`、`playbook.run`。

### 验收

- agent 批量研究必须绑定 plan。
- plan 有最大 run 数和时间预算。
- promote 需要 QA report 和 policy evaluation。
- M6 已存在的 ModelPackage promote 路径改为调用 PromotionPolicy。
- 至少 6 条 playbook 可执行到 backtest compare。

## 13. M10：Production Signal + Paper

### 目标

正式信号和 paper trading 进入 StrategyGraph Runtime，消除与回测口径漂移。

### 范围

- ProductionSignalRun。
- PaperSession 3.0。
- order intent。
- execution diagnostics。
- ReproducibilityBundle 初版。

### 交付物

- 表：`production_signal_runs`、`paper_sessions`、`paper_daily`、`reproducibility_bundles`。
- 服务：`ProductionSignalService`、`PaperTradingService3`、`ReproducibilityService`。
- API/MCP：`signal.generate_production`、`paper.create_session`、`paper.advance`、`repro.export_bundle`。

### 验收

- 只有 validated/published StrategyGraph 能生成正式信号。
- signal 输出 target portfolio 和 order intent。
- paper 与 backtest 使用同一 graph runtime。
- published asset 能导出 reproducibility bundle。

## 14. M11：UI 工作台

### 目标

把 UI 从功能模块导航转成 project/run/artifact/QA/promotion 的研究工作台。

### 范围

- Project-first 导航。
- Research Home。
- Runs & Artifacts。
- QA/Promotion/Cleanup。
- Factor/Dataset/Model/Portfolio/Strategy Labs。
- Signal & Paper。

### 交付物

- 前端 API 类型更新。
- 项目上下文替代全局 market selector。
- run/artifact/lineage 浏览页。
- promote 和 cleanup 操作页。

### 验收

- human 能从 backtest 回溯到 strategy graph、portfolio、model、dataset、factor、data snapshot。
- human 能审阅 QA 并 promote/archive。
- cleanup preview 显示影响范围。

## 15. M12：CN_A 分阶段接入

### 目标

按 profile/rule/policy 接入 A 股能力，不再横向硬套 US 流程。

### 阶段 A：CN_A 数据和资产

- symbol normalization。
- BaoStock provider adapter。
- CN calendar。
- daily bars。
- trade_status。

验收：

- CN_A data.status 可用。
- 能查询 CN_A bars 和 tradability。

### 阶段 B：CN_A Universe 和 Factor subset

- 全 A / 指数 universe。
- ST、停牌、上市天数过滤。
- time_series factor adapter。

验收：

- CN_A universe profile 可用。
- CN_A factor preview/materialize 可用。

### 阶段 C：CN_A 回测规则

- 涨跌停不可成交。
- 停牌不可成交。
- T+1 卖出限制。
- 手数规则。

验收：

- CN_A backtest smoke 可跑。
- execution diagnostics 能解释未成交原因。

## 16. 推荐开发节奏

建议每个里程碑拆成 1-3 个 PR，每个 PR 控制在可 review 范围内。

推荐顺序：

1. Schema + service skeleton。
2. API/MCP thin wrapper。
3. Minimal happy path。
4. Artifact/lineage/QA 补齐。
5. Migration/legacy adapter。
6. UI 接入。
7. E2E 和文档。

每个 PR 必须包含：

- 影响范围。
- 新增/修改表。
- API 变化。
- 兼容策略。
- 验证命令。
- 是否影响 2.0 fallback。

## 17. 验证矩阵

| 类型 | 命令/方式 | 适用阶段 |
| --- | --- | --- |
| 后端 smoke | `uv run python scripts/e2e_demo.py` 或 3.0 e2e | 每个后端里程碑 |
| 前端构建 | `cd frontend && pnpm build` | UI/API 类型变化 |
| 迁移 dry-run | `uv run python scripts/migrate_3_0.py --dry-run` | M0-M3 |
| 迁移 apply | `uv run python scripts/migrate_3_0.py --apply` | M3 后 |
| API 手测 | FastAPI docs 或 MCP tools | 每个 API 里程碑 |
| 数据检查 | data.status、universe.profile、dataset.profile | M2-M4 |
| 策略一致性 | backtest vs production signal same-day explain | M8-M10 |

## 18. 稳定迭代策略

### Feature Flags

建议增加配置：

```yaml
features:
  qagent_3_kernel: true
  qagent_3_market_data: false
  qagent_3_dataset: false
  qagent_3_strategy_runtime: false
  qagent_3_ui: false
```

每个阶段先 behind flag，上线后再默认开启。

### Adapter 策略

- 旧 factor 通过 legacy FactorBase adapter 进入 FactorSpec。
- 旧 strategy 通过 legacy StrategyGraph node 包装。
- 旧 model 先成为 archived/candidate ModelExperiment。
- 旧 backtest 只迁 summary，不承诺完整重放。

### 数据安全

- 3.0 schema 初始化不覆盖 2.0 表。
- 迁移脚本默认 dry-run。
- cleanup 默认只 preview。
- published artifact 默认不可自动删除。

## 19. 里程碑完成定义

一个里程碑完成必须满足：

- schema 已初始化或迁移脚本可重复执行。
- service 层有最小 happy path。
- REST/MCP 不绕过 service。
- run/artifact/lineage 已记录。
- 至少一个 smoke test。
- 文档更新。
- 明确 legacy fallback 是否仍保留。

未满足这些条件，不进入下一个依赖里程碑。

## 20. 第一阶段最小可发布版本

如果需要压缩范围，第一阶段 MVP 可以只包含 M0-M10，CN_A 只做到 profile fixture。

MVP 必须具备：

- US_EQ project。
- ResearchRun/Artifact/Lineage。
- US data migration。
- Universe materialization。
- Dataset materialize/profile。
- Factor preview/materialize/evaluate。
- ModelExperiment/ModelPackage。
- Portfolio/Risk/Execution assets。
- StrategyGraph backtest。
- QA Gate。
- ProductionSignalRun。
- ReproducibilityBundle。

MVP 不必具备：

- 完整 CN_A 数据更新。
- 完整 UI 工作台。
- 所有高级 factor compute mode。
- 高级组合优化器。
- broker integration。
