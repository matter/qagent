# QAgent V3.2 架构迁移与策略模块重构需求及开发计划

> 文档状态：需求与开发计划版  
> 版本定位：V3.2 大版本迭代  
> 当前分支：`3.0`  
> 目标读者：开发 agent、研究 agent、系统维护者、人类验收者  
> 基线文档：`docs/v3.0/3.0-refactor-design.md`、`docs/v3.0/3.0-implementation-roadmap.md`、`docs/v3.1/3.1-strategy-parameter-and-execution-plan.md`

## 1. 版本定位

V3.0 建立了 ResearchRun、Artifact、Lineage、MarketProfile、Dataset、ModelPackage、StrategyGraph、portfolio/risk/rebalance/execution specs 等新架构。V3.1 补入了计划价格成交、T+1 close fallback、策略默认参数和 per-order execution intent 的第一批能力。

V3.2 的目标不是继续横向新增功能，而是完成一次“剥离式大版本迭代”。旧系统只作为迁移输入、对账基线和归档来源，不作为运行时兼容目标。

1. 把旧系统中仍有价值的因子、股票集合、k 线、模型、特征集、策略和模拟交易依赖重新录入、导入或重实现到 3.0 主路径。
2. 删除旧架构运行时、旧入口、旧服务和不可复现过程数据。
3. 重构策略模块，把 alpha、选股、仓位控制、订单意图、执行模拟、状态管理拆成可审计的层。
4. 让策略可以动态决定每笔交易的成交方式，同时由统一执行引擎负责成交、记账、诊断和复现。

## 2. 范围和非范围

### 2.1 范围

- 全面迁移到 3.0 新架构，旧入口不保留业务运行兼容层；迁移完成后删除或禁用旧入口。
- 所有因子以 3.0 `factor_specs` 重新录入；旧源码只作为录入来源。
- 所有股票集合和 k 线数据导入新的 `assets`、`universes`、`daily_bars` 等表结构；旧表只作为导入来源。
- 模拟交易中仍在使用的策略、模型、特征集及其依赖链重建为 StrategyGraph、ModelPackage、Dataset、FeaturePipeline。
- 清理旧架构服务、路由、重复表、临时结果和不可复现冗余内容。
- 重构策略模块，覆盖功能整理、现代订单模型、仓位控制算法、动态执行意图和 human/agent 诊断。
- 把 backlog 中仍有价值的问题纳入 V3.2 计划：状态化长任务、research cache 原子写、DuckDB 单写者防护、agent 研究元数据和多 agent 观测。

### 2.2 非范围

- 不处理数据源质量升级；该方向已按产品决策从 backlog 删除。
- 不接入实盘券商，不做真实下单。
- 不做分钟、Tick、高频、盘口排队撮合。
- 不做多用户权限系统。
- 不把 QAgent 变成 Codex/Paperclip 之类的 agent 控制平面；QAgent 只保存研究计划、证据、任务和产物。
- 不提供旧设计、旧 API、旧服务的长期兼容层；任何需要保留的能力都必须在 3.0 架构中重新实现。
- 不在迁移未完成前删除可验证资产。任何删除都必须有备份、dry-run、对账报告和 rollback 方案。

## 4. 迁移原则

### 4.1 资产优先级

V3.2 把现有内容分为四类：

| 类别 | 处理方式 | 示例 |
| --- | --- | --- |
| 必须保留 | 重新录入、导入或重实现到 3.0；失败要阻断旧资产删除 | 因子源码、股票集合、k 线、活跃模拟交易依赖的策略/模型/特征集 |
| 条件保留 | 若被活跃 paper session、published/candidate 资产或最新研究报告引用，则重建为 3.0 资产；否则归档 | 老模型实验、老回测结果、信号运行记录 |
| 可归档 | 写入 archive 文档或 artifact manifest，不进入主 UI | 历史调参试验、失败任务日志、旧诊断 JSON |
| 可删除 | 备份后删除，不再进入新架构 | 重复缓存、无 lineage 的临时结果、旧架构冗余过程表 |

### 4.2 无损升级要求

- 所有迁移先生成 dry-run manifest，包含表名、行数、主键、内容 hash、依赖关系、迁移动作、风险等级。
- apply 阶段只写 3.0 新表或 archive 表，不覆盖旧表。
- cutover 前允许离线读取旧表做对账，但应用运行时只验证 3.0 新路径；不把双路径作为兼容能力。
- 删除前必须生成 `cleanup_preview`，人类确认后才执行。
- 删除动作必须记录 `migration_run_id`、备份路径、删除列表、恢复命令。
- 迁移脚本必须可重复执行。重复执行只更新同一映射或跳过已完成项，不能产生重复资产。

### 4.3 剥离策略

- REST、MCP、UI 必须继续共用 service layer。
- 旧 API 不承担业务运行兼容。迁移期如必须保留入口，只能返回迁移提示、410/弃用响应，或触发离线迁移工具；不能委托旧服务继续执行。
- 新 API 可以继续把缺省 market 解析为 `US` 作为产品默认值，但这不是旧接口兼容承诺。
- UI 只展示 3.0 工作流；旧页面在迁移完成后删除，不保留只读业务入口。
- MCP 只暴露 3.0 工具；旧工具删除或替换为 3.0 等价能力。

禁止模式：

- 禁止新功能依赖旧 service、旧 schema、旧策略接口或旧回测引擎。
- 禁止通过 adapter 长期包裹旧运行时来伪装成 3.0 能力。
- 禁止为了旧 UI/旧 API 保留双写、双跑、双结果口径。
- 禁止把旧表作为 3.0 运行时查询源；旧表只能在离线迁移和对账时读取。

## 5. 保留和删除策略

### 5.1 因子

保留所有因子，包括 built-in 和 custom factor。

迁移目标：

- 旧 `factors` -> `factor_specs`
- 旧 `factor_values_cache` 只作为校验来源；正式缓存必须重新 materialize，或在完整 metadata 校验通过后导入为 `factor_runs` / `factor_values` / artifact
- 因子源码 snapshot -> artifact 或 `factor_specs.source_code`

允许重录：

- 若旧因子没有完整 metadata，但源码可读，则重录为新的 `FactorSpec`。
- 若旧缓存无法验证来源，只保留源码，缓存作废并在 3.0 下重新 materialize。

验收：

- 因子数量不减少，除非 manifest 标记为重复别名。
- 每个因子有 market、代码 hash、依赖字段、可用性说明。
- 至少选 3 个 US 因子和 3 个 CN 因子完成 preview/materialize/evaluate smoke。

### 5.2 股票集合和 k 线数据

保留所有股票集合和 k 线数据。

迁移目标：

- 旧 `stocks` -> `assets` / `asset_identifiers`
- 旧 `stock_groups` / `stock_group_members` -> `universes` / `universe_memberships`
- 旧 `daily_bars` -> 新 daily bar 表或现有 3.0 daily bar 结构

允许导入：

- 若新表结构需要 `asset_id`，先建立 ticker 到 asset 的稳定映射，再导入 k 线。
- 若 group 是当前快照集合，membership 的 point-in-time 语义标记为 `snapshot_membership`，不伪装成历史成分。

验收：

- 按 market、ticker、date 聚合的 daily bar row count 与迁移前一致。
- 每个 stock group 的成员数与迁移前一致，允许去重后减少但必须在报告中列出。
- US/CN 的 asset_id 映射无跨市场冲突。

### 5.3 模拟交易依赖资产

保留模拟交易中涉及的所有策略、模型和特征集。

迁移顺序：

1. 找出活跃或近期 paper session。
2. 追踪 session 使用的 strategy、model、feature_set、factor、label、universe。
3. 将 feature_set 重建为 `FeaturePipeline` 或 `Dataset`。
4. 将模型重建为 `ModelPackage`，无法重建时至少保存模型文件、训练配置、输入特征列表和 code snapshot。
5. 将策略重写为 StrategyGraph 节点或 3.0 策略源码；旧 `StrategyBase` 代码只作为源码参考，不保留运行 adapter。
6. 创建新的 3.0 paper session 或 paper session mapping。

验收：

- 每个保留 paper session 都能找到 3.0 strategy graph、model package、dataset/feature pipeline 映射。
- 至少一个 US 活跃 paper session 完成迁移源记录与 3.0 新会话的对账报告；差异要可解释，但不保留旧会话运行能力。
- 若 CN 存在活跃 paper session，也要完成同样对照；无 CN 活跃 session 时记录为 N/A。

### 5.4 删除范围

删除或归档对象：

- 无 active/published/candidate 引用的旧回测结果。
- 无模型包或策略图引用的旧 signal runs。
- 可重建的 feature/factor 临时缓存。
- 旧架构中的重复配置表、过程 JSON、失败任务残留。
- 迁移前后完全等价但不再被服务层读取的旧表。

禁止删除对象：

- 原始 k 线。
- 股票集合。
- 因子源码。
- 活跃 paper session 的依赖链。
- published/candidate 模型、策略、dataset、artifact。
- 迁移报告、备份、cleanup preview、rollback 指令。

## 6. V3.2 目标架构

### 6.1 统一运行路径

```text
REST / MCP / React UI
  -> Application Service
  -> Research Kernel
  -> Dataset / Factor / Model / StrategyGraph / Execution / Paper Runtime
  -> DuckDB + Artifact Store
```

旧系统迁移后只允许两种形态：

1. archive / migration evidence：用于审计迁移来源、对账和回滚。
2. removed / disabled：运行时代码、路由、MCP 工具和 UI 入口删除或禁用。

应用运行时不得调用旧 service、旧 backtest engine、旧 signal service 或旧 paper trading service。

### 6.2 统一资产关系

```text
ResearchProject
  -> Universe
  -> FeaturePipeline + LabelSpec
  -> Dataset
  -> ModelExperiment
  -> ModelPackage
  -> StrategyGraph
  -> BacktestRun / ProductionSignalRun / PaperSession
```

所有正式产物必须有：

- `ResearchRun`
- `Artifact`
- `Lineage`
- `lifecycle_stage`
- `market_profile_id`
- `params_snapshot`
- `code_snapshot_refs`
- `data_snapshot_refs`

### 6.3 迁移后的模块边界

| 模块 | 职责 | 不再负责 |
| --- | --- | --- |
| Factor | 因子定义、preview、materialize、evaluate、cache metadata | 训练样本拼接、策略执行 |
| Dataset | 特征/标签面板、profile、sample、QA | 模型算法细节 |
| Model | 基于 Dataset 的实验、包、预测、ranking/listwise 目标 | 策略仓位和执行 |
| Strategy | alpha、选股、状态、订单意图、组合目标 | 成交记账、现金变动、成本核算 |
| Portfolio | 目标权重、约束、仓位控制、再平衡 | alpha 计算 |
| Execution | 订单模拟、成交判定、成本、失败原因 | 选择股票 |
| Paper | 生产信号到模拟账户的推进和审计 | 训练模型、重算因子 |

## 7. 策略模块重构

### 7.1 现有功能整理

当前系统已有能力：

- 旧 `StrategyBase.generate_signals(context)` 返回 `signal/weight/strength`，这是迁移输入，不是 V3.2 目标接口。
- 策略可声明 `default_backtest_config`、`default_paper_config`。
- 策略输出可携带 `execution_model`、`planned_price`、`planned_price_buffer_bps`、`planned_price_fallback`、`price_field`、`time_in_force`、`order_reason`。
- 旧回测已有 position sizing、max positions、rebalance buffer、min holding、cooldown、planned price、close fallback，这些能力需要在 3.0 策略/组合/执行模块中重新实现。
- 3.0 已有 `portfolio_construction_specs`、`risk_control_specs`、`rebalance_policy_specs`、`execution_policy_specs`、`state_policy_specs`、`strategy_graphs`。
- `execution_model_service.py` 已有 `next_open`、`planned_price`、`next_close` 的基础规范。

现有问题：

- 旧策略把 alpha、selection、position、execution intent 混在同一个 DataFrame。
- 3.0 StrategyGraph 与旧 BacktestService 仍有重叠执行逻辑，V3.2 的目标是删除旧执行逻辑，而不是维持两套口径。
- 下单语义还只是 execution_model 字段，没有完整订单对象和生命周期。
- 仓位控制算法散落在回测配置、策略默认参数、portfolio service 中。
- UI 中选项多，但哪些属于实验环境、哪些属于策略默认、哪些属于订单意图不够清晰。

### 7.2 新策略分层

V3.2 的策略模块拆为七层：

| 层 | 名称 | 输入 | 输出 |
| --- | --- | --- | --- |
| 1 | AlphaModel | Dataset、factor、model prediction、state | score、direction、confidence |
| 2 | SelectionPolicy | alpha frame、universe、tradability | candidate set |
| 3 | PortfolioConstruction | candidate、score、constraints | target weights |
| 4 | PositionController | current holdings、target weights、risk、rebalance policy | desired position deltas |
| 5 | OrderIntentGenerator | position deltas、execution preference | order intents |
| 6 | ExecutionSimulator | order intents、daily bars、market rules、cost model | fills、blocks、trade diagnostics |
| 7 | StatePolicy | fills、holdings、strategy state | next state |

策略代码可以选择：

- 只输出 alpha，让系统 portfolio/execution specs 接管后续层。
- 输出目标权重，让 PositionController 和 ExecutionSimulator 接管。
- 输出 order intents，让系统只负责成交、记账、诊断。
- 在高级模式中提供自定义节点，但必须保留标准输入输出 schema。

### 7.3 现代订单模型

V3.2 引入标准订单意图对象 `OrderIntent`。它不是券商订单，而是低频回测和模拟交易中的可审计执行意图。

核心字段：

| 字段 | 说明 |
| --- | --- |
| `order_intent_id` | 稳定 ID |
| `strategy_graph_id` / `strategy_id` | 来源策略 |
| `decision_date` | 决策日期 |
| `execution_date` | 计划执行日期 |
| `market` / `asset_id` / `ticker` | 标的 |
| `side` | buy/sell |
| `target_weight` | 目标权重 |
| `target_quantity` | 目标数量，可由系统推导 |
| `order_type` | market、limit、stop、stop_limit、trailing_stop、close、planned_price |
| `execution_model` | next_open、planned_price、next_close、future_twap、future_vwap 等 |
| `limit_price` | 限价 |
| `stop_price` | 止损/触发价 |
| `planned_price` | 计划成交价 |
| `planned_price_buffer_bps` | 计划价成交缓冲 |
| `planned_price_fallback` | cancel、next_close |
| `time_in_force` | day、gtc_simulated、ioc_simulated |
| `priority` | 同日候选竞争和现金不足时的排序 |
| `order_reason` | 策略原因 |
| `intent_metadata` | 额外诊断 |

V3.2 第一阶段支持：

- `market` / `next_open`
- `close` / `next_close`
- `planned_price`，支持 buffered high/low 和 `next_close` fallback
- `limit` 的日线近似成交规则
- `stop` / `stop_limit` 的日线近似触发规则

后续预留但不强制实现：

- trailing stop
- bracket/OCO
- TWAP/VWAP 日频近似
- volume participation cap
- partial fill
- multi-day simulated GTC

### 7.4 条件单语义

日频条件单必须明确“近似模型”，不能假装知道日内路径。

| 类型 | 日频近似 | 默认限制 |
| --- | --- | --- |
| next_open market | T 决策，T+1 open 成交 | open 缺失则 block |
| next_close close | T 决策，T+1 close 成交 | 不是 T 同日 close |
| planned_price | T+1 `low*(1+buffer) <= price <= high*(1-buffer)` | 失败可 cancel 或 next_close fallback |
| buy limit | 若 T+1 low <= limit_price，则以 limit_price 或更保守价格成交 | 不判断日内先后顺序 |
| sell limit | 若 T+1 high >= limit_price，则以 limit_price 或更保守价格成交 | 不判断日内先后顺序 |
| stop | 若 high/low 触发 stop，则转为 market/close 近似 | 同日 high/low 触发顺序不可知时记录 warning |
| stop_limit | 同时满足触发和 limit 区间才成交 | 不确定路径记录 warning |

所有条件单输出必须记录：

- `fill_status`
- `fill_type`
- `fill_price`
- `blocked_reason`
- `daily_bar_used`
- `path_assumption`
- `warning`

### 7.5 仓位控制模块

仓位控制从“回测参数”升级为 `PositionController` 资产。目标是支持算法迭代，同时避免为了保持比例每天频繁微调。

基础能力：

- `target_weight` 模式。
- `target_quantity` 模式。
- `equal_weight`、`score_proportional`、`inverse_vol`、`risk_parity_lite`、`custom_weight`。
- 单票上限、行业/集合上限、现金保留、最大持仓数。
- turnover budget。
- lot size / 最小交易单位。
- 最小交易金额 `min_trade_value`。
- 最小权重变化 `min_weight_delta`。
- 再平衡 band / deadband。
- min holding days、max holding days、reentry cooldown。
- forced exit：风险违反、模型失效、停牌后恢复、人工禁买。

避免每日微调的规则：

- 非强制交易只有在 `abs(target_weight - current_weight) >= rebalance_band` 时才生成订单。
- 同时满足 `trade_value >= min_trade_value` 和 `abs(delta_weight) >= min_weight_delta` 才交易。
- 对持仓内股票可使用更宽的 reduce/add band，例如 `rebalance_buffer_add` 与 `rebalance_buffer_reduce`。
- 若目标持仓集合不变且漂移在 band 内，则只记录 drift，不下单。
- 若现金不足，按 priority、forced_exit、risk_reduce、new_entry 的顺序分配。

### 7.6 策略动态控制成交方式

策略可以在每个信号或每笔订单中选择成交方式：

```python
return pd.DataFrame(
    {
        "signal": [1, -1],
        "weight": [0.08, 0.0],
        "strength": [0.92, 0.40],
        "execution_model": ["planned_price", "next_close"],
        "order_type": ["planned_price", "close"],
        "planned_price": [101.5, None],
        "planned_price_buffer_bps": [50, None],
        "planned_price_fallback": ["next_close", None],
        "time_in_force": ["day", "day"],
        "priority": [80, 100],
        "order_reason": ["gap-sensitive entry", "risk exit"],
    },
    index=["AAPL", "MSFT"],
)
```

约束：

- 策略只表达意图，不直接修改现金、持仓、成交价和收益。
- execution engine 负责验证字段、填充默认值、成交判定、成本、失败原因。
- 所有动态字段必须进入 order intent snapshot，供 backtest/signal/paper 共用。
- 不允许策略在 T 日使用 T+1 high/low/close 生成 planned_price 或执行方式。

### 7.7 策略配置归属

配置分三类：

| 类别 | 归属 | 示例 |
| --- | --- | --- |
| 实验环境 | run/session | market、universe、start/end、capital、benchmark、commission、slippage、debug |
| 策略默认 | strategy metadata | max_positions、rebalance band、min holding、execution preference、planned_price_fallback |
| 每笔订单意图 | strategy output / OrderIntentGenerator | order_type、execution_model、planned_price、limit_price、stop_price、priority |

合并顺序：

```text
system defaults < strategy defaults < run/session override < per-order intent
```

结果必须持久化：

- `strategy_defaults_snapshot`
- `run_overrides_snapshot`
- `effective_config`
- `config_provenance`
- `order_intents_snapshot`

## 8. 数据库升级和回滚方案

### 8.1 迁移表和映射

新增或完善迁移控制表：

- `migration_runs`
- `migration_asset_map`
- `migration_keep_delete_manifest`
- `migration_validation_results`
- `cleanup_previews`
- `cleanup_actions`

`migration_asset_map` 至少包含：

- `migration_run_id`
- `source_table`
- `source_id`
- `new_table`
- `new_id`
- `asset_kind`
- `action`
- `status`
- `content_hash_before`
- `content_hash_after`
- `notes`

### 8.2 升级流程

1. 停止写入型后台任务。
2. 执行 `scripts/backup_data.sh`。
3. 运行 V3.2 baseline：health、DB preflight、旧资产 inventory smoke、3.0 smoke、frontend build。
4. dry-run 资产扫描，生成 keep/delete manifest。
5. 人类确认 manifest。
6. apply 迁移，只写 3.0 表和 archive/manifest。
7. 跑 row count/hash/lineage validation。
8. 开启 3.0-only flags，让 UI/API/MCP 只走 3.0。
9. 运行 3.0-only 验收，不再依赖旧运行时。
10. 生成 cleanup preview。
11. 人类确认后删除冗余旧表、旧路由、旧服务和旧 UI。

### 8.3 回滚

- cutover 前：删除新写 3.0 迁移资产，回到迁移前备份状态；这只是恢复环境，不是保留旧系统兼容运行。
- cutover 后、cleanup 前：关闭 3.0-only flags，恢复备份或重新执行迁移；不能通过旧 adapter 继续提供业务能力。
- cleanup 后：从备份恢复完整数据库，或按 `cleanup_actions` 的 rollback 指令恢复 archive，再重新执行迁移。

任何 rollback 都必须记录到 `migration_runs.status` 和 `migration_validation_results`。

## 12. 风险和决策点

### 12.1 风险

- 迁移误删资产：用 dry-run、manifest、人类确认和备份降低风险。
- 迁移口径差异：每个切换点必须做源记录与 3.0 新结果对账，不要求逐笔完全相同，但差异要可解释。
- 策略自由度过大：通过标准 OrderIntent schema 和 ExecutionSimulator 约束副作用。
- 条件单日频近似被误读：所有条件单必须记录 path assumption 和 warning。
- DuckDB 单写者限制：V3.2 只做操作防护，不把数据库替换作为本版本目标。

### 12.2 需要人类确认的点

- 哪些 paper session 算“必须保留”。
- 旧回测和旧 signal 结果归档多久。
- 旧 UI 删除时点，以及是否需要单独导出审计报告替代只读入口。
- planned_price 是否在 V3.2 后切为新策略默认模式。
- 条件单第一阶段是否只做 planned/limit/stop，还是加入 bracket/OCO 的诊断-only 支持。

## 13. 完成定义

V3.2 完成时应满足：

- backlog 中数据源质量需求已删除，非数据源剩余问题要么关闭，要么明确降级为后续低优先级。
- 因子、股票集合、k 线和活跃模拟交易依赖资产已完整重新录入、导入或重实现到 3.0。
- 旧架构与 3.0 并存造成的核心语义重复被消除，默认且唯一研究路径走 3.0。
- 策略模块支持标准订单意图、动态成交方式、条件单日频近似、仓位控制算法和防微调规则。
- human 可以在 UI 中量化验收迁移、策略、订单、仓位、paper 运行状态。
- agent 可以通过文档、REST/MCP 和 artifact 完成全流程研究，不需要读 chat history 才知道系统规则。
