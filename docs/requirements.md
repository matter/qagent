# QAgent - 低频量化交易研究系统 需求规格说明书

> 版本：v2.2 | 基于 `target.md` 原始设想，经专业评审、决策确认、四维落地审查后整理。

---

## 1. 系统定位与边界

### 1.1 核心定位

**低频量化策略研究系统**，覆盖从因子挖掘到信号生成的完整研究链路。

- 面向**日频**周期（日线数据，日级调仓）
- 首期覆盖**美股市场**，架构预留多市场扩展能力
- 定位为**研究工具**，非交易执行系统

### 1.2 使用模式

单用户系统，无需认证/权限体系。支持三种交互模式：

| 模式 | 说明 | 接口 |
|---|---|---|
| 人工操作 | 用户通过浏览器完成研究全流程 | Web UI |
| Agent 自主 | LLM Agent 程序化调用系统能力 | MCP Server |
| 人机协作 | 用户通过 Agent 辅助完成研究任务 | MCP Server（Agent 代理操作） |

### 1.3 系统边界

**做：**
- 日线数据获取、存储、管理
- 因子研究、评价、管理
- 特征工程与 ML 模型训练
- 策略编写、回测、评价
- 交易信号生成与导出
- 行情与研究结果可视化

**不做：**
- 券商对接 / 实盘交易下单
- 高频 / Tick 级 / 分钟级数据
- 多用户 / 权限管理
- 舆情数据（列为后续扩展）

## 2. 产品原则

> 以下原则贯穿系统所有模块，优先级高于功能数量。

### 2.1 时序正确性优先

首期宁可少做功能，也必须确保：
- **不使用未来数据**——信号基于 T 日收盘生成，T+1 开盘执行
- **不忽略退市股票**——保留退市数据防止幸存者偏差（首期在数据源允许范围内）
- **不使用错误的验证方式**——禁止随机 K-Fold，强制时序交叉验证
- **不忽略交易成本**——所有回测必须包含佣金和滑点

### 2.2 资产化而非脚本化

系统的核心不是"帮用户跑一次脚本"，而是把研究成果沉淀为可追溯的资产。因子、特征集、模型、策略、回测报告和信号记录都必须可保存、可复用、可追溯。不可接受的实现方式：
- 因子只保存名字，不保存代码和版本
- 模型只保存二进制文件，不保存特征定义和预处理参数
- 回测只展示图，不保存配置和输入条件

### 2.3 人与 Agent 共用一套内核

UI、CLI、MCP 只是入口形式，底层能力共享同一套服务层。否则人和 Agent 会走两套逻辑，结果不一致、维护成本翻倍。

### 2.4 本地优先

系统默认本地部署、本地存储、本地运行。运行时不依赖网络（仅数据更新时需要）。这与单用户研究工具定位一致。

## 3. 核心研究流程

> 以下定义系统支持的 7 条研究流程。每条流程描述用户的操作路径和系统的自动化行为。

### 流程 1：因子研究

```
提出假设 → 编写因子脚本 → 指定回测区间和股票池 → 系统计算因子值
→ 系统生成评价报告（IC/IR/分组收益等） → 用户查看报告
→ [不满意] 修改因子脚本 → 重新计算（快速迭代循环）
→ [满意] 保存因子至因子库（含源码、参数、评价快照）
```

**关键设计：** 因子迭代是最高频的操作。系统必须支持**快速重算**——修改因子代码后，秒级看到评价结果更新。实现方式：因子脚本热加载 + 增量计算（仅重算变更部分）。

### 流程 2：特征集构建

```
浏览因子库 → 查看各因子评价指标 → 查看因子间相关性矩阵
→ 圈选多个低相关因子 → 配置预处理方式 → 保存为特征集
```

**关键设计：** 圈选因子时系统自动展示候选因子间的**相关性矩阵**，警告高相关因子对（|corr| > 0.7），避免引入冗余特征。

### 流程 3：模型训练

```
选择特征集 → 选择学习目标（如未来5日收益率） → 选择模型类型
→ 配置模型超参数 → 配置训练方式（滚动窗口/扩展窗口 + 时间划分）
→ 执行训练（异步，可查看进度） → 查看评估结果（含特征重要性）
→ [可选] 对比多个模型的结果
→ 保存模型（模型文件 + 特征定义 + 预处理参数，确保可复现预测）
```

**关键设计：** 训练可能耗时数分钟，必须**异步执行**并展示进度。训练完成后自动生成特征重要性排序。

### 流程 4：策略构建与回测

```
基于策略模板编写 Python 策略脚本 → 策略内引用因子值和/或模型预测分数
→ 定义买卖信号逻辑 → 选择仓位管理算法 → 配置回测参数（成本、基准、区间）
→ 执行回测 → 查看回测报告（净值曲线、指标、交易流水）
→ [可选] 多策略对比
→ 保存策略
```

### 流程 5：信号生成

```
选择已保存策略 → 确认数据已更新至最新 → 选择股票池
→ 系统自动执行完整管线：计算所需因子 → 运行模型预测 → 执行策略逻辑
→ 输出当日交易信号列表 → 导出 CSV/JSON
```

**关键设计：** 信号生成时，系统自动处理上游依赖（因子计算 → 预处理 → 模型预测 → 策略执行），用户无需手动逐步触发。如果数据未更新到最新交易日，系统应**警告**而非静默使用旧数据。

### 流程 6：行情浏览

```
搜索/选择股票 → 查看日线 K 线图（主图：蜡烛+均线，副图：成交量+MACD）
→ 切换时间范围 → 叠加自定义因子曲线（可选） → 叠加买卖信号标记（可选）
```

### 流程 7：数据管理

```
查看数据状态（覆盖范围、最后更新时间、数据质量统计）
→ 执行增量更新 / 全量重建
→ 管理股票分组（创建、编辑、删除分组；手动选股或条件筛选）
```

## 4. 数据层

### 4.1 行情数据

#### 4.1.1 数据内容

| 数据项 | 说明 |
|---|---|
| 日线 OHLCV | Open, High, Low, Close, Volume |
| 复权因子 | 用于计算前复权/后复权价格 |
| 未复权价格 | 保留原始价格，复权通过复权因子动态计算 |
| 股票基础信息 | 代码(ticker)、名称、交易所(NYSE/NASDAQ)、行业分类(GICS sector)、状态(活跃/退市) |
| 指数数据 | SPY（标普500 ETF）作为默认基准，可扩展 |

**复权策略：** 存储未复权原始数据 + 复权因子。前复权价格在查询时动态计算。原因：前复权价格会随新的公司行动（拆股、分红）发生变化，动态计算避免全量历史重算。

#### 4.1.2 股票池范围

- NYSE + NASDAQ 活跃标的（约 5000~8000 只）
- **保留退市股票历史数据**，防止幸存者偏差
- 股票状态标记：活跃 / 退市 / 停牌

**股票列表来源：** yfinance 本身不提供完整股票列表。首期方案：从 NASDAQ 官方 FTP（`nasdaqtrader.com`）获取 NYSE + NASDAQ 股票列表（免费、官方），仅行情数据通过 yfinance 获取。后续可切换至 Polygon.io 统一获取。

#### 4.1.3 数据更新

| 操作 | 说明 |
|---|---|
| 增量更新 | 从最后已有日期更新至最新交易日 |
| 全量初始化 | 首次部署时拉取全部历史（建议最近 10 年） |
| 数据修复 | 检测并修补缺失数据段 |
| 状态查询 | 最后更新日期、覆盖股票数、数据缺失率 |
| 可中断续传 | 大批量更新中断后，可从上次断点继续 |

**更新时机：** 美股收盘后（美东 16:00，约北京时间 04:00/05:00）。系统不自动定时更新（避免后台任务复杂度），由用户或 Agent 主动触发。

#### 4.1.4 增量更新策略（最小化 API 请求）

核心原则：**本地数据优先，只请求本地没有的部分。**

**增量更新算法：**

```
1. 读取本地每只股票最后已有数据日期（DuckDB 聚合查询，毫秒级）
2. 获取当前最新交易日（通过交易日历计算，无需网络请求）
3. 过滤出需要更新的股票列表（最后日期 < 最新交易日的股票）
4. 对需要更新的股票，只请求 [最后日期+1, 最新交易日] 区间的数据
5. 按批次下载并写入 DuckDB
```

**请求量估算（日常增量更新）：**
- 每日新增数据：~5000 股票 × 1 天 = 5000 行
- yfinance 批量下载：`yf.download(tickers, start=last_date, end=today)` 可一次请求多只
- 按 500 只/批，仅需 ~10 个 HTTP 请求即可完成全市场日更
- 对比全量更新（5000 股 × 10 年）的 ~10 批，日常更新请求量极小

**全量初始化优化：**
- 首次拉取 5000~8000 只股票 × 10 年历史是最大请求量场景
- 采用分批 + 进度持久化：每完成一批立即写入 DuckDB 并记录进度
- 中断后重启：从进度文件恢复，跳过已完成的批次
- 估计耗时：~500 只/批 × 16 批 × 3 秒间隔 ≈ 1~2 分钟（不含下载时间）

#### 4.1.5 yfinance 防限流策略

Yahoo Finance 未公布官方限流阈值，社区实测约为 **~2000 请求/小时/IP**。以下策略确保稳定运行：

| 策略 | 实现 |
|---|---|
| **批量下载** | 使用 `yf.download(tickers_list)` 多股票模式，500 只/批为最佳平衡点 |
| **随机延迟** | 批次间 2~3 秒基础延迟 + 0.5~1.5 秒随机抖动，避免固定模式被检测 |
| **指数退避** | 遇到 429/错误时：5s → 10s → 20s → 40s → 最大 120s |
| **自定义 Session** | 注入 `requests.Session` 带 retry adapter（3 次重试，自动处理 429/500/502/503） |
| **requests-cache** | 使用 `requests_cache` 库缓存 HTTP 响应，相同参数的重复请求直接命中本地缓存 |
| **User-Agent 轮换** | 设置合理的 User-Agent 头，避免被识别为脚本 |
| **失败隔离** | 单只股票失败不阻断整个批次，记录失败列表，全部批次完成后统一重试失败项 |
| **可中断续传** | 每批完成后持久化进度，kill 进程后可从断点继续 |

```python
# 示例：防限流配置
DOWNLOAD_CONFIG = {
    "batch_size": 500,                    # 每批股票数
    "batch_delay_base": 2.0,              # 批次间基础延迟（秒）
    "batch_delay_jitter": (0.5, 1.5),     # 随机抖动范围
    "retry_count": 3,                     # 单批最大重试次数
    "retry_backoff_base": 5.0,            # 重试退避基础秒数
    "retry_backoff_max": 120.0,           # 最大退避秒数
    "on_429_delay": 60.0,                 # 收到 429 后等待秒数
    "use_cache": True,                    # 启用 requests-cache
    "cache_ttl_hours": 6,                 # 缓存过期时间
}
```

#### 4.1.6 数据质量校验

每次数据更新后自动执行：
- 检测价格异常跳变（单日涨跌幅 > 50% 非拆股情况）
- 检测成交量异常（为 0 或缺失）
- 检测时间序列断裂（非停牌的连续缺失）
- 生成数据质量报告

#### 4.1.7 交易日历

使用 `exchange_calendars` 库管理美股交易日历（NYSE 日历），用于：
- 判断某日是否为交易日
- 计算 N 个交易日前/后的日期
- 数据更新时确认预期记录数
- 后续 A 股接入时切换为 SSE 日历

### 4.2 数据存储

**选型：DuckDB**

| 特性 | 匹配度 |
|---|---|
| 列式存储，分析查询快 | 因子计算需要按列批量读取，完美匹配 |
| 单文件部署，零运维 | 单用户本地系统，无需数据库服务 |
| 原生 SQL + Python API | 降低开发门槛 |
| 原生 Parquet 支持 | 便于数据导入导出 |

**并发访问策略：** DuckDB 采用单写多读模型。本系统通过**单进程架构**规避并发问题——FastAPI 进程同时承载 Web UI 后端和 MCP Server（SSE 传输），所有数据库访问通过同一进程内的服务层串行化写操作。

**存储结构：**

```
data/
├── qagent.duckdb          # 主数据库（行情、元数据、因子值缓存）
├── models/                # 模型文件（pickle/joblib）
│   └── {model_id}/
│       ├── model.joblib
│       └── metadata.json  # 特征定义、预处理参数、评估指标
├── factors/               # 因子 Python 源码
│   └── {factor_name}/
│       └── v{version}.py
├── strategies/            # 策略 Python 源码
│   └── {strategy_name}/
│       └── v{version}.py
└── update_progress.json   # 数据更新断点续传进度文件
```

### 4.3 股票分组管理

| 功能 | 说明 |
|---|---|
| 手动分组 | 用户手动选择股票加入分组 |
| 条件筛选 | 基于表达式自动筛选（如 `sector == 'Technology' AND avg_volume > 1000000`） |
| 分组用途 | 回测股票池、因子计算范围、信号生成范围 |
| 内置分组 | 系统预置 "全市场"、"标普500成分股" 等常用分组 |

#### 4.3.1 股票池时点一致性（防前视偏差）

**问题：** 如果用户今天定义了一个"市值 Top 500"分组，然后用这个分组回测过去 5 年——相当于用了未来信息（今天的市值排名）去筛选历史标的，构成前视偏差。

**处理方式：**
- **静态分组（手动选股）：** 回测时直接使用固定股票列表，用户自行对前视偏差负责。系统在回测报告中标注"使用固定股票池，未做时点还原"。
- **条件筛选分组：** 首期同样使用当前时点的筛选结果。系统在回测报告中明确警告此限制。
- **后续优化：** 实现按历史时点动态还原股票池（需要历史成分股数据支持）。

这是一个有意识的首期简化，系统必须在回测结果中明确标注，而不是静默忽略。

## 5. 因子研究模块

### 5.1 因子协议

所有因子必须实现统一接口：

```python
class FactorBase(ABC):
    """因子基类"""
    name: str                    # 因子名称
    description: str             # 因子描述
    params: dict                 # 因子参数（如窗口期）
    category: str                # 类别：momentum / volatility / volume / fundamental / custom

    @abstractmethod
    def compute(self, data: pd.DataFrame) -> pd.DataFrame:
        """
        输入：面板数据 DataFrame
              index = DatetimeIndex（交易日）
              columns = MultiIndex(field, ticker) 或 ticker
              field 包含: open, high, low, close, volume
        输出：因子值 DataFrame
              index = DatetimeIndex
              columns = ticker
              values = 因子值（float，允许 NaN）
        """
        pass
```

**设计原则：**
- 因子只接受行情数据作为输入，不允许直接依赖其他因子的输出（避免隐式依赖链，保持每个因子独立可计算）
- 若需组合因子能力，在**特征集**层面完成（如将多个因子的值组合、变换）
- 因子计算必须是**时点确定性的**——给定相同输入数据，输出完全一致

### 5.2 因子类型

| 类型 | 计算维度 | 示例 |
|---|---|---|
| 时序因子 | 单股票沿时间轴计算 | 20日动量、波动率、MACD |
| 截面因子 | 同一时间点跨股票计算 | 市值排名、行业相对强弱 |
| 面板因子 | 同时涉及时序和截面 | 行业中性动量（先算动量再截面去均值） |

### 5.3 内置因子与技术指标库

#### 5.3.1 技术指标库集成

系统集成成熟的三方技术指标库，因子编写时可直接调用，无需重复实现基础指标。

**主库：TA-Lib**（C 编译核心，性能最优，行业标准 20+ 年）
**辅助库：pandas-ta**（纯 Python，补充 TA-Lib 未覆盖的指标）

| 库 | 指标数 | 性能 | 安装 | 用途 |
|---|---|---|---|---|
| **TA-Lib** | ~158 | 极快（C 核心） | 需安装 C 依赖：`brew install ta-lib`（macOS）或 `conda install -c conda-forge ta-lib` | 主力，所有核心指标 |
| **pandas-ta** | ~130 | 良好（向量化） | `pip install pandas-ta`（纯 Python） | 补充 TA-Lib 缺少的指标 |

**架构模式：** 系统提供统一的指标适配层，封装 TA-Lib 和 pandas-ta 调用，输入输出统一为 Pandas Series/DataFrame：

```python
# 用户在因子脚本中直接调用
from qagent.indicators import ta

rsi = ta.rsi(close, period=14)           # 自动路由到 TA-Lib
macd, signal, hist = ta.macd(close)       # TA-Lib 实现
adx = ta.adx(high, low, close, period=14) # TA-Lib 实现
zscore = ta.zscore(close, period=20)      # 自定义实现
```

**趋势类：**

**波动率类：**

| 因子 | 基于 | 说明 |

**统计类：**

| 因子 | 基于 | 说明 |
|---|---|---|
| `ZScore_N` | 自定义 | N 日价格 Z 分数，均值回归 |
| `LinReg_Slope_N` | `ta.linearreg_slope()` | 线性回归斜率，趋势速率 |
| `Turnover_N` | 自定义 | N 日平均换手率 |

### 5.4 因子评价体系

#### 5.4.1 单因子评价指标

| 指标 | 计算方法 | 判断标准 |
|---|---|---|
| **IC（信息系数）** | 每期因子值与下期收益的 Spearman 秩相关 | \|IC均值\| > 0.03 有效 |
| **IR（信息比率）** | IC均值 / IC标准差 | \|IR\| > 0.5 较好 |
| **IC 胜率** | IC > 0 的期数占比 | > 55% 较稳定 |
| **IC 衰减曲线** | IC 随持有期（1日、5日、20日...）的变化 | 判断因子时效性 |
| **分组收益** | 按因子值分 5 组，计算各组平均收益 | 组间单调递增/递减 |
| **多空收益** | Top组 - Bottom组的收益 | 衡量因子区分度 |
| **因子换手率** | 相邻期因子排名变化幅度 | 过高意味着交易成本高 |
| **因子覆盖率** | 有效因子值（非NaN）占比 | < 80% 需警惕 |

#### 5.4.2 因子评价报告输出

- IC 时序折线图 + IC 均值/IR 统计
- 分组累计收益曲线（5组）
- 因子值分布直方图（按时间抽样展示）
- 换手率时序图
- 汇总统计表

### 5.5 因子库管理

| 功能 | 说明 |
|---|---|
| 保存 | 记录：名称、描述、源码、参数、分类、评价指标快照、创建时间 |
| 版本 | 同名因子支持多版本，版本号自增，可查看版本历史 |
| 缓存 | 因子值按 (因子版本, 股票池, 日期范围) 缓存至 DuckDB，避免重复计算 |
| 失效 | 因子代码更新时，自动标记旧版本缓存为失效 |
| 浏览 | 列表展示所有因子，支持按分类筛选、按 IR 排序 |
| 删除 | 删除因子及关联缓存（如被特征集引用则阻止删除） |

## 6. 特征工程模块

### 6.1 特征集定义

特征集 = 一组因子 + 预处理配置。

```
FeatureSet:
  name: str
  description: str
  factors: List[FactorRef]        # 引用的因子（名称+版本）
  preprocessing: PreprocessConfig  # 预处理配置
  created_at: datetime
```

### 6.2 构建流程

1. 从因子库浏览和筛选候选因子
2. 系统展示候选因子间的**相关性矩阵热力图**
3. 高相关因子对（|corr| > 0.7）标红警告
4. 用户确认选择后，配置预处理参数
5. 保存特征集

### 6.3 预处理流水线

预处理按以下固定顺序执行，每步可配置开关：

| 步骤 | 方法 | 说明 |
|---|---|---|
| 1. 缺失值处理 | 前值填充 / 截面均值填充 / 丢弃 | 默认：前值填充 |
| 2. 去极值 | MAD (中位数绝对偏差, 3倍) / Winsorize (1%/99%) | 默认：MAD |
| 3. 标准化 | Z-score / Rank（截面排名归一化到0~1） | 默认：Rank |
| 4. 中性化（可选） | 对市值/行业做回归取残差 | 默认：关闭 |

**关键约束：** 预处理参数（如 Z-score 的均值和标准差）在训练时**从训练集计算并保存**，预测时直接使用保存的参数，而非重新计算。这是防止未来信息泄露的关键。

## 7. 模型训练模块

### 7.1 模型接口

采用 sklearn 兼容接口，首期实现 LightGBM：

```python
class ModelBase(ABC):
    """模型基类，兼容 sklearn 接口"""

    @abstractmethod
    def fit(self, X: pd.DataFrame, y: pd.Series, **kwargs) -> 'ModelBase':
        pass

    @abstractmethod
    def predict(self, X: pd.DataFrame) -> pd.Series:
        pass

    @abstractmethod
    def get_params(self) -> dict:
        pass

    def feature_importance(self) -> pd.Series:
        """返回特征重要性排序，部分模型可能不支持"""
        raise NotImplementedError
```

首期实现：
- `LightGBMModel`：封装 LightGBM 的回归和分类

后续可扩展（接口已就绪）：
- `XGBoostModel`
- `LinearModel`（Ridge / Lasso）
- `RandomForestModel`

### 7.2 学习目标

| 目标 | 任务类型 | 说明 | 适用场景 |
|---|---|---|---|
| **未来 N 日收益率** | 回归 | 最直接，但受离群值影响 | 通用 |
| **未来 N 日截面排名** | 回归（0~1） | 排名值更稳定 | 截面选股 |
| **涨跌方向** | 二分类 | 简化问题 | 择时 |
| **超额收益（vs SPY）** | 回归 | 剔除 beta 后的 alpha | 对冲策略 |

用户选择目标类型和参数（如 N=5），系统自动生成 label 列。

### 7.3 训练方式

**严格约束：禁止随机交叉验证。** 时序数据只能使用时序感知的验证方式。

| 方式 | 说明 | 适用 |
|---|---|---|
| **单次划分** | Train / Valid / Test 按时间顺序三段划分 | 快速验证 |
| **滚动窗口（Rolling）** | 固定长度训练窗口向前滚动 | 检验稳定性 |
| **扩展窗口（Expanding）** | 训练窗口起点固定，终点向前扩展 | 最大化训练数据 |

滚动/扩展训练的参数：
- `train_window`：训练窗口长度（交易日数）
- `valid_window`：验证窗口长度
- `step`：每次滚动步长
- `purge_gap`：训练集和验证集之间的隔离期（防止信息泄露，默认 5 个交易日）

### 7.4 训练输出

| 输出 | 说明 |
|---|---|
| 评估指标 | 回归：IC, RMSE, R²；分类：AUC, 准确率, F1 |
| 特征重要性 | 柱状图，按重要性排序 |
| 预测值 vs 实际值 | 散点图（回归）/ 混淆矩阵（分类） |
| 分组收益 | 按预测值分组的收益表现（同因子评价逻辑） |
| 滚动性能 | 各滚动窗口的性能变化趋势（如有） |

### 7.5 模型保存

模型保存为一个自包含的目录，包含所有复现预测所需的信息：

```
models/{model_id}/
├── model.joblib              # 模型权重（或滚动训练的多个模型）
├── metadata.json             # 以下所有配置的 JSON 序列化
│   ├── feature_set_id        # 关联的特征集
│   ├── feature_names          # 特征名有序列表
│   ├── preprocessing_params   # 预处理统计量（均值、标准差等）
│   ├── model_type             # 模型类型标识
│   ├── model_params           # 模型超参数
│   ├── train_config           # 训练方式配置
│   ├── label_config           # 学习目标配置
│   ├── eval_metrics           # 评估指标
│   └── created_at             # 创建时间
```

## 8. 策略与回测模块

### 8.1 策略协议

```python
class StrategyBase(ABC):
    """策略基类"""
    name: str
    description: str

    @abstractmethod
    def generate_signals(self, context: StrategyContext) -> pd.DataFrame:
        """
        输入 context 包含：
          - prices: 行情数据
          - factor_values: dict[factor_name -> DataFrame]  已计算的因子值
          - model_predictions: dict[model_id -> Series]    模型预测分数
          - current_positions: DataFrame                   当前持仓（回测用）

        输出 DataFrame:
          index = ticker
          columns = [signal, weight, strength]
          signal: 1(买入) / -1(卖出) / 0(无操作)
          weight: 目标权重 (0~1)，由仓位管理算法填充
          strength: 信号强度（用于排序和仓位分配）
        """
        pass

    def required_factors(self) -> List[str]:
        """声明策略依赖的因子（系统自动计算）"""
        return []

    def required_models(self) -> List[str]:
        """声明策略依赖的模型（系统自动运行预测）"""
        return []
```

**策略依赖声明：** 策略通过 `required_factors()` 和 `required_models()` 显式声明依赖。系统在回测/信号生成前自动按依赖顺序计算因子值和模型预测。用户无需手动管理管线。

### 8.2 仓位管理

仓位管理作为独立组件，与策略信号解耦：

| 算法 | 说明 | 首期 |
|---|---|---|
| **等权（EqualWeight）** | 所有入选股票等比例分配 | ✅ 实现 |
| **信号加权（SignalWeight）** | 按信号强度分配权重 | ✅ 实现 |
| **最大持仓约束（MaxPosition）** | 单只股票权重上限 + 最大持仓数 | ✅ 实现 |
| 风险平价（RiskParity） | 按波动率倒数加权 | 后续 |

仓位管理算法可组合使用（如：信号加权 + 最大持仓约束）。

### 8.3 回测引擎

#### 8.3.1 回测执行时点（核心约束）

**回测口径必须显式定义，这是量化研究正确性的基石。**

采用保守且无未来函数的口径：

```
T 日收盘后：使用 T 日及之前的所有数据计算因子、运行模型、生成信号
T+1 日开盘：按 T+1 日开盘价执行交易
```

| 时点 | 说明 |
|---|---|
| 信号生成 | T 日收盘后，使用 T 日收盘价及历史数据 |
| 信号执行 | T+1 日开盘价成交 |
| 成本计算 | 在 T+1 开盘价基础上叠加佣金和滑点 |

**禁止的做法：** 使用 T 日收盘数据计算信号并在 T 日收盘成交——这构成未来函数（look-ahead bias），因为收盘价在收盘时才确定，不可能在收盘前基于收盘价下单。

信号生成模块在生成"实际可执行"信号时也遵循此口径：使用最新可用收盘数据生成信号，信号预期在下一个交易日开盘执行。

#### 8.3.2 回测方式

采用**向量化回测**，自研实现。

选择自研而非使用第三方库（如 vectorbt、backtrader）的原因：
- vectorbt 开源版已停止维护（2023年起），Pro 版为付费闭源
- backtrader 自 2020 年起未更新，事件驱动架构不适合因子截面策略
- 日频调仓的向量化回测逻辑简洁（核心代码量可控，约 1~2 周开发量）
- 需要与本系统的因子/模型管线深度集成
- 完全掌控回测细节（成本模型、执行时点、信号处理）

#### 8.3.3 回测配置

| 参数 | 说明 | 默认值 |
|---|---|---|
| `initial_capital` | 初始资金 | 1,000,000 |
| `start_date` / `end_date` | 回测区间 | 最近 5 年 |
| `benchmark` | 基准标的 | SPY |
| `universe` | 股票池（分组名） | 全市场 |
| `rebalance_freq` | 调仓频率 | 每日 |
| `commission_rate` | 佣金费率（双边） | 0.001 (0.1%) |
| `slippage_rate` | 滑点模型（按比例） | 0.001 (0.1%) |
| `max_positions` | 最大持仓数 | 50 |
| `position_sizing` | 仓位管理算法 | 等权 |

#### 8.3.4 回测评价指标

**收益指标：**

| 指标 | 说明 |
|---|---|
| 年化收益率 | 复合年化增长率 (CAGR) |
| 累计收益率 | 期末/期初 - 1 |
| 超额收益（年化） | 策略年化 - 基准年化 |

**风险指标：**

| 指标 | 说明 |
|---|---|
| 年化波动率 | 日收益率标准差 × √252 |
| 最大回撤 | 历史最大峰谷跌幅 |
| 最大回撤持续期 | 从峰值到恢复的最长天数 |

**风险调整指标：**

| 指标 | 说明 | 参考值 |
|---|---|---|
| **夏普比率（Sharpe）** | (年化收益 - 无风险利率) / 年化波动率 | > 1.0 较好 |
| **卡尔玛比率（Calmar）** | 年化收益 / 最大回撤 | > 1.0 较好 |
| **索提诺比率（Sortino）** | (年化收益 - 无风险利率) / 下行波动率 | > 1.5 较好 |

**交易指标：**

| 指标 | 说明 |
|---|---|
| 总交易次数 | 买入+卖出次数 |
| 胜率 | 盈利交易占比 |
| 盈亏比 | 平均盈利 / 平均亏损 |
| 年化换手率 | 年均换手倍数 |
| 总交易成本 | 佣金 + 滑点的累计金额 |

#### 8.3.5 回测输出

- 净值曲线（策略 vs 基准，对数坐标可选）
- 回撤曲线
- 月度收益热力图（年 × 月 矩阵）
- 年度收益柱状图
- 持仓数量变化曲线
- 交易流水明细表（可导出 CSV）
- 指标汇总卡片

### 8.4 策略对比

支持选择多个已保存策略，在相同回测参数下并排对比：
- 净值曲线叠加
- 指标对比表格
- 相关性分析（策略收益相关性）

### 11.4 统一任务执行框架

因子计算、模型训练、回测和信号生成均通过统一的任务执行框架运行（Web UI 和 MCP 共享同一套框架），而非由各入口自行拼接调用。

**任务框架职责：**

| 职责 | 说明 |
|---|---|
| 标准化输入输出 | 每类任务有明确的输入参数和输出结构 |
| 状态管理 | 排队中 → 运行中 → 完成 / 失败 |
| 日志记录 | 每个任务自动记录开始时间、结束时间、参数、结果摘要 |
| 产物保存 | 任务产出（因子值、模型文件、回测报告）自动归档 |
| 异常捕获 | 用户代码异常不影响系统进程，错误信息完整返回 |
| 进度反馈 | 长任务可报告进度百分比（Web UI 展示进度条，Agent 可轮询） |

这一框架是"人与 Agent 共用一套内核"原则（§2.3）的落地保障。

**CPU 密集操作的异步处理：**

FastAPI 运行在 asyncio 事件循环上。因子计算、模型训练等 CPU 密集操作如果直接在 `async def` 中执行，会阻塞事件循环，导致 UI API 和 MCP SSE 连接同时冻结。

处理方式：
- 短操作（< 1s）：定义为普通 `def`（FastAPI 自动放入线程池执行）
- 长操作（> 1s）：通过 `asyncio.run_in_executor(executor, fn)` 放入独立线程池
- DuckDB 写操作：通过 `threading.Lock` 串行化，防止多线程并发写入冲突

**MCP 集成架构示例：**

```python
# 单进程内 FastAPI + MCP 共存
from fastapi import FastAPI
from mcp.server.fastmcp import FastMCP
import duckdb

app = FastAPI()
mcp_server = FastMCP("qagent")
db = duckdb.connect("data/qagent.duckdb")  # 进程内共享单连接

# Web UI API 路由
@app.get("/api/stocks/{ticker}")
async def get_stock(ticker: str): ...

# MCP 工具注册
@mcp_server.tool()
def get_stock_data(ticker: str, start_date: str, end_date: str): ...

# MCP 挂载为 FastAPI 子应用
app.mount("/mcp", mcp_server.streamable_http_app())
```

这一架构确保 Web UI 和 Agent 共享同一个 DuckDB 连接、同一套服务层逻辑，消除数据不一致风险。

## 15. 数据源抽象

### 15.1 DataProvider 接口

```python
class DataProvider(ABC):
    """数据源抽象接口"""

    @abstractmethod
    def get_stock_list(self) -> pd.DataFrame:
        """获取股票列表：ticker, name, exchange, sector, status"""
        pass

    @abstractmethod
    def get_daily_bars(self, tickers: List[str], start: date, end: date) -> pd.DataFrame:
        """获取日线数据：date, ticker, open, high, low, close, volume, adj_factor"""
        pass

    @abstractmethod
    def get_index_data(self, symbol: str, start: date, end: date) -> pd.DataFrame:
        """获取指数/ETF 数据（基准用）"""
        pass
```

### 15.2 首期实现：YFinanceProvider

- 股票列表：从 `nasdaqtrader.com` FTP 获取（官方、免费、覆盖 NYSE+NASDAQ）
- 日线数据：通过 yfinance 批量下载（`yf.download()` 多股票模式）
- 复权数据：使用 `auto_adjust=False` 获取原始 OHLC + Adj Close，通过 `Adj Close / Close` 推算复权因子
- 限流策略：批次请求，每批 100 只，间隔 1~2 秒，避免触发 Yahoo 反爬
- 错误重试：单只股票失败不阻断整体更新，指数退避重试 3 次，记录失败列表待后续重试

**已知限制（必须了解）：**

| 限制 | 影响 | 应对措施 |
|---|---|---|
| 不提供股票列表 API | 无法从 yfinance 获取完整 ticker 列表 | 用 nasdaqtrader.com FTP 独立获取 |
| 退市股数据不完整 | 许多退市股返回空数据或 404 | 首期接受此限制，在回测报告标注；后续切换 Polygon.io |
| Ticker 被复用风险 | 退市后 ticker 被新公司使用，返回新公司数据 | 数据校验时检测上市日期突变 |
| 无官方 SLA | Yahoo 可能随时改变接口，导致 yfinance 失效 | DataProvider 抽象层保证可快速切换数据源 |
| 复权因子偶有延迟 | 拆股/分红后 1~2 天内 Adj Close 可能未更新 | 数据质量校验检测异常复权因子跳变 |
| 大规模请求被限流 | 8000+ 股票全量更新可能触发 429 | 批次请求 + 退避策略 + 可中断续传 |

## 16. 错误处理策略

| 场景 | 处理方式 |
|---|---|
| 数据源请求失败 | 重试 3 次（指数退避），仍失败则记录日志、跳过该标的继续 |
| 因子计算异常 | 捕获异常，该因子标记为计算失败，返回 NaN DataFrame，不阻断其他因子 |
| 因子值全为 NaN | 警告用户，因子可能逻辑有误或数据不足 |
| 模型训练失败 | 记录完整错误栈，任务标记为失败，通知用户 |
| DuckDB 写入失败 | 事务回滚，记录日志，提示用户检查磁盘空间 |
| 用户因子/策略代码错误 | 捕获异常，展示完整 traceback，不影响系统稳定性 |
